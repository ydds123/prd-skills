## 维度说明

B端产品的核心价值在于��撑业务流程的高效运转。流程设计和角色权限设计是B端产品的两大支柱。《决胜B端》第6章详细介绍了UML建模方法在流程设计中的应用，跨部门流程图（泳道图）是产品经理使用频率最高的��具。

**核心理论框架：**

1. **UML流程建模**（《决胜B端》6.3节）：产品���理常用的UML图包括ER图、跨部门流程图（泳道图）、状态机图、活动图、用例图。其中跨部门流程图"使用频率最高"。

2. **跨部门流程图/泳道图**（《决胜B端》6.3.2节）：用泳道区分不同角色或部门，在泳道内描述每个角色的操作步骤，泳道之间的连线描述角色间���信息传递和任务流转。

3. **状态机图**（《决胜B端》6.3.3节��：描述��务对象在不同状态之间的流转，包括状态名称、触发事件、转换条件。适用于订单、工单、审批单等有生命周期的业务对象。

4. **角色权限矩阵**（《决胜B端PRD模板》第12节）：以表格��式精确到页面元素级别定义每个角色的访问和操作权限。

5. **RBAC权限模型**（《决胜B端》5.3节）：基于角色的访问控制（Role-Based Access Control），通过"用户-角色-权限"的三层结构管理系统权限。

## 检查清单

### 8.1 业务流程设计

- [ ] **核心流程的可视化**
  - 检查内容：核心业务流程是否用流程图可视化呈现
  - 优质标准：使用跨部门泳道图呈现核心流程，泳道按角色或部门划分，每个步骤有明确的开始和结束条件，流程中的判断分支有清晰的条件标注
  - 问题标志：只用文字描述流程而无可视化图表，或使用简单的线性流程图而未区分角色
  - 参考依据：《决胜B端》6.3.2节——跨部门流程图是产品经理使用频率最高的UML图

- [ ] **流程的完整性**
  - 检查内容：流程是否覆盖了从业务发起到业务结束的完整路径
  - 优质标准：
    - 有明���的流程起点和终点
    - 覆盖正常路径和主要异常路径
    - 每个判断节点都有"是"和"否"两个分支
    - 流程中的等待、超时、撤回等情况有处理方案
  - 问题标志：流程只画了正常路径（Happy Path）；判断节点只有一个分支；流程有开始无结束
  - 参考依据：《决胜B端》流程设计要求——覆盖正常流程、异常流程、边界条件

- [ ] **流程的层次性**
  - 检查内容：复杂流程是否采用了分层描述（总体流程→子���程）
  - 优质标准：先用一张总体流程图展示主干（5-7个关键步骤），再对复杂步骤展开子流程详细描述；总体流��和子���程之间有清晰的引用关系
  - 问题标志：一张流程图试图展示所有细节，密密麻麻无法阅读；或只有高层流程缺少关键子流程
  - 参考依据：《决胜B端》"自顶向下"的设计思路

- [ ] **审批流程设计**
  - 检查内容：涉及审批的业务是否有完整的审批流程设计
  - 优质标准：定义了审批节点、审批人（角色/个人/条件触发）、���批动作（通过/驳回/转交/加签）、审批超时处理、审批记录保留
  - 问题标志：提到需要审批但未设计审批流程细节（如谁审批、驳回后如何处理）
  - 参考依据：《决胜B端PRD模板》10.2.1.1节业务流程图示例

### 8.2 状态管理设计

- [ ] **状态机图的完整性**
  - 检查内容：有状态生命周期的业务对象是否都有状态机图
  - 优质标准：每个有状态的实体（订单、工单、审批单、客户等）都有对应的状态机图，包含所有状态、所��转换路径、转换��件和触发事件
  - 问题标志：有状态实体但无状态机图，或状态机图不完整（如只有正向流转而无逆向流转如退回、撤销）
  - 参考依据：《决胜B端》6.3.3节状��机图

- [ ] **状态命名的清晰性**
  - 检查内容：状态名称是否清晰无歧义，对用户有意义
  - 优质标准：状态名称使用业务语言而非技术语言（如用"待审批""已发货"而非"状态1""状态2"）；相似状态有明确区分（如区分"已提交"和"审批中"）
  - 问题标志：状态名称使用编号或技术术语；多个状态容易混淆
  - 参考依据：B端产品设计最佳实践

- [ ] **异常状��处理**
  - 检查内容：是否考虑了状态流转中的异常情况
  - 优质标准：设计了撤回（发起人撤回）、驳��（审批人驳回）、超时自动处理、异常中止、手动干预等异常状态处理；状态机中无"死状态"
  - 问题标志：状态只能正向流转不能撤回或驳回；某些状态到达后无法继续流转
  - 参考依据：《决胜B端》异常处理设计

### 8.3 角色与权限设计

- [ ] **角色体系设计**
  - 检查内容：是否设计了完整的角色体系
  - 优质标准：列出所有系统角色、每个角色的定义和职责范围、角色之间的层级关系（如管理员>运营人员>普通用户）、是否支持自定义角色
  - 问题标志：角色定义模糊或过于笼统（只有"管理员"和"普通用户"）
  - 参考依据：《决胜B端》5.3节RBAC权限模型

- [ ] **权限矩阵的精细度**
  - 检查内容：权限设计是否精确到页面元素级别
  - 优质标准：以矩阵表格形式列出每个角色在每个页面、每个按钮/操作上的权限；区分了查看权限、编辑权限、删除权限、审批权限等
  - 问题标志：权限只到菜单级别（如"管理员可以访问所有菜单"），未精确到按钮和操作级别
  - 参考依据：《决胜B端PRD模板》第12节��色权限矩阵——精确到"页面元素"级别

- [ ] **数据权限设计**
  - 检查内容：除了功能权限���是否考虑了数据权限（谁能看到什么数据）
  - 优质标准：定义了数据权限范围（如"区域经理只能看到本区域数据""员工只能看到自己负责的客户"），数据权限与组织架构关联
  - 问题标志：所有角色看到相同数据，未考虑数据隔离需求
  - 参考依据：《决胜B端》权限管理——功能权限和数据权限是两个独立维度

- [ ] **超级管理员与系统管理功能**
  - 检查内容：���否设计了系统管理类功能（用户管理���角色管理、系统配置等）
  - 优质标准：有超级管理员角色负责用户增删改、角色分配、系统参数配置等管理功能
  - 问题标志：没有系统管理功能设计，不知道谁来管理用户和角色
  - 参考依据：B端产品��准功能模块

### 8.4 组织架构与多租户

- [ ] **组织架构模型**
  - 检查内容：是否设计了与业务匹配的组织架构模型
  - 优质标准：支持层级式组织架构（如公司→区域→门店），组织层级与数据权限、审批流���关联；如果是SaaS产品，支持多租户的组织���构隔离
  - 问题标志：需要组织架构��未设���，或组织架构不支持业务实际的层级关系
  - 参考依据：《决胜B端》1.3.4节Org组织架构���理平台

- [ ] **多租户设计**（SaaS产品必查）
  - 检查内容：SaaS产品是否有多租户设计，确保租户之间的数据和功能隔离
  - 优质标准：明确了租户隔离级别（数据库级/Schema级/表级/行级），租户之间完全不可见对方数据；租户管理员可以独立��理本租户的用户和权限
  - 问题标志：SaaS产品无多租户设计概念
  - 参考依据：《决胜B端》SaaS产品设计要点

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 流程只有文字无流程图 | 缺乏可视化建模能力 | 用ProcessOn等工具绘制跨��门泳道图 |
| 流程只有正常路径 | 未���异常场景推演 | 对每个判断节点做"如果否/如果异常"推演 |
| 状态只能正向不能逆向 | 状态机设计不完整 | 补充撤回、驳回、超时等逆向和异常状态 |
| 权限过于粗放 | 未做精细化权限矩阵 | 按模板建立角色-页面-元素权限矩阵 |
| 所有人看到同样数据 | 缺少数据权限设计 | 设计基于组织架构的数据权限体系 |
| 审批流程不完整 | 只考虑了通过路径 | 补充驳回、��交、超时、撤回等完整审批动作 |
