# MVP策略与演进蓝图检查

> Supporting reference for the `check-prd` skill. 在执行 MVP 策略与演进蓝图检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品建设是一个长期过程，不可能一步到位。MVP（最小可行产品）策略和产品演进蓝图决定了产品如何从0到1、从1到N逐步成长。《决胜B端》强调产品建设应"先做加法再做减法"——先充分理解所有业务场景，再有策略地裁剪出MVP。

**核心理论框架：**

1. **MVP设计原则**（《决胜B端》第2章和第4章）：MVP不是"最简单的版本"，而是能够验证核心假设、支撑核心业务闭环的最小功能集。产品定位明确后，需要规划"功能模块与演进蓝图"。

2. **先做加法再做减法**（《决胜B端》5.1节）："我们做产品设计，重要的是能够想清楚所有的业务场景和问题，然后基于研发资源和实现周期，做出一个综合评估后的合理方案"。

3. **种子客户策略**（《决胜B端》3.1节）：商业化产品在MVP阶段需要选择"有代表性的标杆客户"进行充分调研和验证，MVP版本应围绕种子客户的核心需求刻画。

4. **产品化策略**（《决胜B端》第8章）：从项目制到产品化的演进路径——先解决单一客户问题，再抽象为标准化产品，最后建立配置化/插件化的产品平台。

5. **迭代与版本管理**（《决胜B端》第10章）：通过需求优先级管理、版本规划、需求池管理等手段持续迭代产品。

## 检查清单

### 11.1 MVP范围定义

- [ ] **MVP功能范围的明确性**
  - 检查内容：是否明确定义了MVP版本包含哪些功能、不包含哪些功能
  - 优质标准：列出MVP的功能清单，对每个功能说明"为什么在MVP中"（必要性）或"为什么不在MVP中"（延后理由）；MVP功能构成完整的业务闭环
  - 问题标志："什么都要做"没有裁剪；或MVP功能不构成闭环（如有下单但没有支付，有创建但没有查询）
  - 参考依据：《决胜B端》4.3节功能模块与演进蓝图

- [ ] **MVP的核心假设验证**
  - 检查内容：MVP是否围绕核心假设设计，上线后能验证哪些关键假设
  - 优质标准：明确列出MVP要验证的核心假设（如"目标客户愿意为X功能付费""Y流程能替代现有Excel方案"），以及如何通过数据验证这些假设
  - 问题标志：MVP设计仅基于功能裁剪而非假设验证导向
  - 参考依据：《决胜B端》产品建设思维——产品经理必须"为价值交付负责"

- [ ] **MVP的业务闭环完整性**
  - 检查内容：MVP是否覆盖了核心业务流程的完整闭环
  - 优质标准：沿核心业务流程走查，每个关键环节都有对应功能（即使是简化版本）；关键的数据流转不中断
  - 问题标志：MVP只有业务流程的前半段（如只有创建无管理，只有录入无报表），用户无法基于MVP完成完整业务操作
  - 参考依据：《决胜B端》产品建设流程——先梳理核心业务流程，再裁剪为MVP

### 11.2 演进蓝图规划

- [ ] **版本演进路线图**
  - 检查内容：是否规划了从MVP到成熟版本的演进路线图
  - 优质标准：有3-4个阶段的版本规划（如V1.0核心流程→V1.5体验优化→V2.0扩展功能→V3.0平台化），每个阶段有明确的功能范围和目标
  - 问题标志：只有V1.0设计，完全没有后续演进考虑；或规划过于远大（一上来就画3年蓝图但不务实）
  - 参考依据：《决胜B端》第4章产品整体方案——"功能模块与演进蓝图"

- [ ] **功能优先级矩阵**
  - 检查内容：非MVP功能是否有优先级排序和分期计划
  - 优质标准：使用优先级框架（如价值/成本矩阵、KANO模型、ICE评分）对功能进行优先级排序，高优先级功能在早期版本实现
  - 问题标志：所有功能同等重要无优先级，或优先级排序缺乏依据
  - 参考依据：《决胜体验设计》引用KANO模型——区分基本型需求、期望型需求、兴奋型需求

- [ ] **架构预留与技术债控制**
  - 检查内容：MVP是否在数据模型和架构层面为后续版本预留了扩展空间
  - 优质标准：MVP的数据模型虽然功能精简但结构完整，不会因为后续版本需要大规模重构数据库；关键扩展点有明确标注
  - 问题标志：MVP采用"临时方案"实现，明确会在后续版本全面重构
  - 参考依据：《决胜B端》5.1节——"一旦表结构定型，以后再想修改，是非常麻烦的事情"

### 11.3 种子客户策略

- [ ] **种子客户选择**（商业化产品必查）
  - 检查内容：是否选择了合适的种子客户来验证MVP
  - 优质标准：种子客户有代表性（代表目标客群的典型特征）、有配合意愿、痛点明确；选择了1-3个种子客户
  - 问题标志：无种子客户计划，打算MVP做完直接面向市场推广
  - 参考依据：《决胜B端》3.1节——"针对一个有代表性的标杆客户进行充分调研，透彻理解业务，并刻画出MVP版本"

- [ ] **从项目到产品的演进策略**（商业化产品必查）
  - 检查内容：是否规划了从服务单一客户（项目制）到标准化产品的演进路径
  - 优质标准：明确了第一阶段为种子客户做定制化实施，第二阶段提炼共性需求标准化，第三阶段建立配置化平台的演进策略
  - 问题标志：商业化产品但始终按单一客户定制化方式运作，无产品化计划
  - 参考依据：《决胜B端》第8章产品化——标准化和个性化之间的平衡

### 11.4 上线与运营准备

- [ ] **上线计划**
  - 检查内容：是否有MVP的上线计划，包括灰度策略和回滚方案
  - 优质标准：定义了上线范围（全量/灰度/AB测试）、上线时间表、数据迁移方案、回滚计划、上线后的监控指标
  - 问题标志：无上线计划，或计划"一次性全量上线"而无灰度策略
  - 参考依据：B端产品上线最佳实践

- [ ] **培训与推广计划**
  - 检查内容：是否规划了用户培训和系统推广方案
  - 优质标准：有用户培训方案（培训材料、培训方式、培训时间表）；有推广方案（如何让用户从旧系统/旧方式切换到新系统）
  - 问题标志：系统做好了但没人用，因为缺乏培训和推广
  - 参考依据：《决胜B端PRD模板》第13节运营计划——"需要执行的推广计划、部署计划、执行计划"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| MVP什么都要做 | 未做功能裁剪，缺乏聚焦 | 以核心假设验证为导向裁剪功能 |
| MVP功能不构成闭环 | 裁剪时只考虑功能复杂度不考虑业务完整性 | 沿核心流程走查确保闭环 |
| 无演进蓝图 | 只关注当前版本缺乏长期规划 | 规划3-4个阶段的版本路线图 |
| 数据模型需要频繁重构 | MVP建模过于简化 | MVP阶段就做完整的数据建模，功能可精简但模型要全面 |
| 做了没人用 | 缺少推广和培训规划 | 在PRD中加入运营和培训计划 |
| 一直做项目制无法标准化 | 无产品化策略 | 明确从项目到产品的三阶段演进 |
