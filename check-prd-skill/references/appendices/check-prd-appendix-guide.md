## 第二部分：检查执行指南

### 2.1 检查前的准备工作

**检查人需要准备：**
1. 完整的PRD文档（最新版本）
2. 如果有相关的原型/设计稿，一并获取
3. 了解PRD的类型：新产品设计/迭代需求/模块需求
4. 了解产品属性：自研系统/商业化SaaS/项目制

**根据PRD类型调整检查侧重：**

| PRD类型 | 侧重维度 | 可降低要求的维度 |
|---------|---------|----------------|
| 新产品完整PRD | 所有维度同等重要 | 无 |
| 迭代需求PRD | 05文档结构、07数据、08流程、09交互、12异常、14运营 | 01业务（简要即可）、03定位（引用上位文档）、10商业、11MVP |
| 模块级需求PRD | 07数据、08流程、09交互、12异常 | 01业务、02类型、03定位、06架构、10商业、11MVP、14运营 |
| SaaS新产品PRD | 所有维度+强化10商业、11MVP、14运营 | 无 |
| 企业自研系统PRD | 01业务、04场景、07数据、08流程、14运营 | 03定位（竞品分析可简化）、10商业（ROI替代市场分析） |

### 2.2 检查执行流程

**标准流程（完整检查，约60-90分钟）：**

1. **快速浏览**（10分钟）
   - 通读PRD全文，获取整体印象
   - 判断PRD类型和产品属性
   - 标注第一印象中明显的问题

2. **重大风险项快速排查**（5分钟）
   - 逐一检查R1-R8风险项
   - 如有发现，记录并继续（不提前终止，完成全面检查）

3. **14维度逐一检查**（40-60分钟）
   - 按维度顺序逐一执行检查清单
   - 每个检查项标注"发现问题""未发现问题""不适用"
   - 有问题的检查项记录具体问题描述和严重程度

4. **汇总与报告**（10-15分钟）
   - 汇总各维度关键发现
   - 汇总问题清单并排优先级
   - 撰写Top 10改进建议

**快速流程（快速检查，约20-30分钟）：**

1. 检查重大风险项（R1-R8）
2. 检查P0级问题对照表
3. 检查P1级问题对照表
4. 快速浏览P2/P3问题对照表
5. 输出简要检查结论

### 2.3 检查执行指导原则

1. **按PRD类型调整预期**：迭代需求PRD不需要完整的商业分析，但仍然需要清晰的需求背景和目标
2. **看整体不纠结个别**：如果某个维度整体质量高但个别检查项缺失，不必特别强调
3. **区分"缺失"和"不适用"**：某些检查项对特定产品不适用（如非SaaS产品不需要多租户设计），应标注"不适用"
4. **亮点也要记录**：检查不只是找问题，优秀的设计也值得记录和学习
5. **建议要可操作**：改进建议应该具体、可执行，而非笼统的方向性建议

### 2.4 检查结果的沟通

**沟通原则：**
1. **先说亮点再说问题**：建立积极的沟通氛围
2. **问题分级呈现**：按P0-P3分级，让PRD作者知道什么最紧急
3. **提供改进方向而非批评**：每个问题都附上参考依据和改进建议
4. **区分"必须改"和"建议改"**：给PRD作者明确的行动指引
5. **可以引用方法论**：引用《决胜B端》《决胜体验设计》中的方法论帮助PRD作者理解为什么这是个问题

**检查报告的传递：**
- P0和否决项问题：当面沟通 + 书面记录
- P1问题：书面记录 + 约定修复时间
- P2/P3问题：书面记录，PRD作者自行安排改进
