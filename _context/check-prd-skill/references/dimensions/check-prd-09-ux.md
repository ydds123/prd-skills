# 交互设计质量检查

> Supporting reference for the `check-prd` skill. 在执行交互设计质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品的用户体验长期被忽视，但随着行业成熟，体验已成为产品竞争力的重要组成。《决胜体验设计》系统性地阐述了B端产品的体验设计方法论，从设计价值观到具体的交互范式，提供了完整的设计指导框架。

**核心理论框架：**

1. **体验设计四层模型**（《决胜体验设计》1.1节）：界面设计UI → 用户体验设计UX → 客户体验设计CX → 服务设计SD。四个层次逐层递进，B端产品经理至少要掌握UI和UX两层。

2. **设计价值观体系**（《决胜体验设计》第6章）：设计价值观 → 设计原则 → 设计范式。价值观是交互设计的灵魂，是对具体设计产出的评判标准。

3. **尼尔森十大可用性原则**（《决胜体验设计》6.2节引用）：系统状态可见性、系统与现实世界的匹配、用户控制与自由、一致性与标准化、错误预防、识别而非回忆、灵活高效、美观简洁、帮助用户识别诊断和恢复错误、帮助文档。

4. **用户体验五要素**（Jesse James Garrett，《决胜体验设计》引用）：战略层 → 范围层 → 结构层 → 框架层 → 表现层，从抽象到具体的五个层次。

5. **消息系统设计**（《决胜体验设计》8.4节）：推送人、推送对象、推送内容、推送渠道四要素；五个推送渠道（站内信、移动端Push、第三方APP、邮件、短信）；消息分级管理。

6. **新用户引导设计**（《决胜体验设计》8.3节）：Onboarding上手引导旅程设计，包括分步骤引导、功能发现、首次使用体验优化。

## 检查清单

### 9.1 设计价值观与原则

- [ ] **设计价值观的一致性**
  - 检查内容：PRD中的交互设计是否体现了一致的设计价值观
  - 优质标准：整个产品的交互风格统一——术语一致、操作范式一致、反馈方式一致、视觉风格一致。如果有设计系统或组件库引用，明确说明
  - 问题标志：同一产品中不同模块的交互风格迥异，似乎由不同人独立设计
  - 参考依据：《决胜体验设计》第6章——优秀的产品都应该有明确的设计价值观

- [ ] **一致性与标准化**（尼尔森原则）
  - 检查内容：相同操作在不同页面是否保持一致的交互方式
  - 优质标准：列表页的查询、筛选、排序操作在所有模块保持一致；表单的校验、保存、取消逻辑统一；操作反馈（Toast/弹窗/页面跳转）在相似场景下保持一致
  - 问题标志：同样的操作在不同页面有不同的交互方式（如有的列表支持搜索有的不支持，有的删除有二次确认有的直接删除）
  - 参考依据：尼尔森第四原则——一致性与标准化

### 9.2 信息架构与导航

- [ ] **信息架构的合理性**
  - 检查内容：产品的信息架构是否符合用户的心智模型
  - 优质标准：功能菜单按用户的业务逻辑组织（如按业务阶段、按角色职能），而非按技术模块组织；一级导航不超过7个（符合米勒法则7±2）；高频功能在醒目位置
  - 问题标志：菜单组织按技术模块而非用户视角；一级导航超过10个；高频功能隐藏在三级菜单中
  - 参考依据：《决胜体验设计》第5章信息架构——"让用户理解软件，找到所需"

- [ ] **导航设计的清晰度**
  - 检查内容：用户是否总能知道自己"在哪里"和"能去哪里"
  - 优质标准：有面包屑导航或层级标题指示当前位置；菜单的选中状态明确；全局导航在所有页面可用；提供搜索或快捷入口帮助用户快速跳转
  - 问题标志：用户点击几次后不知道自己在哪；返回路径不清晰；菜单选中状态缺失
  - 参考依据：尼尔森第一原则——系统状态可见性

### 9.3 核心交互设计

- [ ] **列表页设计规范**
  - 检查内容：列表页是否包含标准的交互元素
  - 优质标准：有查询条件区（支持简单查询和高级查询切换）；有列表展示区（支持排序、分页、列定制）；有批量操作区；每行有明确的操作入口（查看/编辑/删除等）；空状态有友好提示
  - 问题标志：列表页缺少查询功能；无分页或分页不合理；无空状态设计
  - 参考依据：《决胜B端PRD模板》10.2节列表页标准描述格式

- [ ] **表单设计规范**
  - 检查内容：表单页是否遵循良好的表单设计原则
  - 优质标准：必填项有明确标识（如红色星号）；字段有合理的排列顺序（重要字段在前）；有实时校验和错误提示；支持暂存/草稿；长表单有分步骤或分组设计；提交后有明确反馈
  - 问题标志：所有字段平铺无分组；无必填标识；校验信息不明确；长表单不支持暂存
  - 参考依据：《决胜体验设计》1.1.1节表单填写体验案例——百字段表单不支持暂存是典型反面案例

- [ ] **操作反馈设计**
  - 检查内容：用户操作后是否有明确的反馈
  - 优质标准：每个操作都有反馈（加载中状态、成功提示、失败原因说明）；关键操作有二次确认（如删除）；异步操作有进度指示；错误信息具体且可操作
  - 问题标志：操作后无反馈（用户不知道是否成功）；错误提示不具体（如只说"操作失败"不说原因）
  - 参考依据：尼尔森第九原则——帮助用户识别、诊断和恢复错误

- [ ] **错误预防设计**
  - 检查内容：是否有机制预防用户犯错
  - 优质标准：危险操作（删除、重置、批量修改）有二次确认；有输入校验和约束（如日期选择器限制不合理的日期范围）；有操作历史和撤销机制；自动保存草稿防止数据丢失
  - 问题标志：危险操作无确认直接执行；无输入约束导致用户容易填错
  - 参考依据：尼尔森第五原则——错误预防优于错误恢复

### 9.4 消息与通知设计

- [ ] **消息系统设计**
  - 检查内容：是否设计了系统的消息通知体系
  - 优质标准：
    - 定义了消息类型和分级（紧急/重要/普通/低优先级）
    - 定义了推送渠道策略（哪类消息走站内信、哪类走Push、哪类走邮件/短信）
    - 消息内容有概要信息而非空洞标题
    - 消息不滥用，有合理的发送频率控制
  - 问题标志：无消息设计；或消息被滥用（什么都推送）；或消息不分级（重要消息被淹没）；或消息文案不描述概要信息
  - 参考依据：《决胜体验设计》8.4节消息系统设计——推送人、推送对象、推送内容、推送渠道四要素；站内信、移动端Push、第三方IM、短信/电话、邮件五种渠道

- [ ] **新用户引导（Onboarding）**
  - 检查内容：如果是SaaS产品或新系统，是否设计了新用户上手引导
  - 优质标准：有分步骤引导流程帮助新用户完成首次关键操作；引导可跳过且可重新触发；引导结束后用户能独立使用核心功能
  - 问题标志：新用户进入空白系统无任何引导，不知道从何开始
  - 参考依据：《决胜体验设计》8.3节新用户引导设计——Salesforce等SaaS产品的Onboarding最佳实践

### 9.5 移动端体验（如适用）

- [ ] **移动端适配设计**
  - 检查内容：如果产品包含移动端，是否有针对移动场景的交互设计
  - 优质标准：移动端不是PC端的简单缩小版，而是根据移动场景重新设计（如支持手势操作、适配单手操作、考虑弱网环境、设计离线能力）
  - 问题标志：移动端设计照搬PC端布局，未考虑移动场景特性
  - 参考依据：《决胜B端》1.1.4节移动互联网时代——"如何在一块小小的屏幕内设计体验良好的App"

### 9.6 组件级交互设计分析（必做专项）

> 本节是本维度最重要的分析，**不能跳过**。前面 9.1-9.5 是合规检查，本节是深度评审。

**执行方法：**

第一步：从 PRD 中提取所有界面交互点，建立清单。包括但不限于：
- 所有列表页、详情页、表单页（新建/编辑）
- 所有弹窗、抽屉、浮层、Toast
- 所有重要操作（提交、删除、审批、导出、批量操作等）
- 所有状态切换（Tab切换、步骤切换、状态流转）

第二步：对每个交互点，评估以下组件选择决策：

**① 弹出层类型选择**（最常见的错误来源）
- **模态弹窗（Modal）**：适合需要用户立即处理、不能忽略、且操作简单的场景（如删除确认、提交确认）。不适合：内容复杂的表单、需要对照列表数据操作、用户可能频繁打开的场景
- **抽屉（Drawer/Slide-over）**：适合需要对照背景数据的详情查看或编辑、操作较复杂但不需要强制打断工作流的场景（如查看订单详情时参考列表）
- **跳转到新页面**：适合表单字段多（>10项）、操作复杂、需要完整视野的场景（如复杂合同录入）
- **非模态提示（Toast/Snackbar）**：适合操作成功/失败反馈，不应用模态弹窗来反馈普通操作结果

**② 表单交互模式**
- 长表单（>15字段）：应分步骤（Wizard）或分组折叠，而非平铺
- 频繁编辑的字段：考虑行内编辑（Inline Edit）代替跳转编辑页
- 关联选择：字段值多（>20项）应用搜索选择（Select + Search）而非普通下拉；存在强关联关系的字段（如省市区三级联动）是否设计了联动逻辑

**③ 列表交互模式**
- 详情查看方式：点击行→抽屉（推荐，保留列表上下文）vs 点击行→跳转详情页（适合详情内容极多），PRD 选择是否与场景匹配
- 批量操作：是否设计了批量选中后的操作栏（浮动工具栏 vs 顶部操作区）
- 空状态：是否设计了空列表引导（带操作按钮的空状态 vs 纯文字提示）

**④ 操作反馈类型匹配**
- 轻量操作成功（如点赞、收藏）：Toast 足够，不需要弹窗
- 重要操作成功（如提交审批、下单）：Toast + 页面状态更新，或带操作入口的通知（如"审批已提交，查看进度→"）
- 危险操作（删除、重置）：必须有模态二次确认，且确认按钮为危险色（红色），而非直接执行
- 异步操作（导入、导出、批量处理）：需要进度指示 + 完成通知，不应让用户面对白屏等待

**第三步：输出组件级分析结论**，格式如下：

```
【组件级问题】PRD第X节 - [功能名]
  当前设计：[描述PRD中的交互方式]
  问题：[为什么不合适]
  建议改为：[具体替代方案]
  原因：[一句话说明更合适的理由]
```

**注意：** 如果 PRD 对某个界面的交互方式描述不详细，也要指出"第X节未说明XX的交互方式，建议补充说明是弹窗还是跳转页，会直接影响研发实现"。

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 不同模块交互风格不统一 | 缺少设计价值观和设计系统 | 建立统一的设计原则和组件库 |
| 菜单层级过深 | 按技术模块而非用户视角组织 | 按业务流程或用户角色重新组织 |
| 表单体验差 | 未遵循表单设计最佳实践 | 添加分组、暂存、实时校验、进度指示 |
| 操作后无反馈 | 遗漏反馈设计 | 为每个操作设计加载、成功、失败三种反馈 |
| 消息推送被用户忽视 | 消息滥用不分级 | 建立消息分级和渠道策略 |
| 新用户不知如何开始 | 缺少Onboarding设计 | 设计分步骤引导和空状态提示 |
