# 产品类型适配性检查

> Supporting reference for the `check-prd` skill. 在执行产品类型适配性检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品种类繁多，不同类型的产品在设计方法、运营模式、技术架构上差异巨大。PRD必须明确产品所属类型，并采用与之匹配的设计策略。错误的产品类型认知会导致设计方向偏差，例如把工具型产品当成业务型产品来设计，会导致过度复杂化。

**核心理论框架：**

1. **B端产品四大分类**（《决胜B端》1.3节）：业务型产品（ERP、CRM等）、工具型产品（电子签名、文档协同等）、交易型产品（电商系统、OMS等）、基础服务产品（权限管理、消息推送等）。四类产品的设计重点和方法论完全不同。

2. **产品部署模式分类**（《决胜B端》1.2.2节）：私有化部署 vs 云部署（SaaS），两种模式对产品化、定制化、安全性要求截然不同。

3. **自研系统 vs 商业化产品**（《决胜B端》第2章）：企业自研自用的系统和商业化对外售卖的产品，在设计思维、客户关系、产品化策略上有根本区别。

4. **B端产品的核心特征**（《决胜B端》1.2.2节）：目标用户是一个群体、用户和客户是不同的人、效能第一体验第二、强调抽象和逻辑、效果受多因素影响。

## 检查清单

### 2.1 产品类型识别

- [ ] **产品类型明确声明**
  - 检查内容：PRD是否明确声明了产品属于哪种类型（业务型/工具型/交易型/基础服务型）
  - 优质标准：在PRD开头或概述部分明确说明产品类型，并解释选择该分类的理由（如"本产品为业务型CRM系统，支撑销售团队的客户管理和商机跟进流程"）
  - 问题标志：PRD通篇没有明确产品定位，或定位模糊（如"一个帮助企业提效的平台"）
  - 参考依据：《决胜B端》1.3节四大产品分类

- [ ] **产品类型与设计方法匹配**
  - 检查内容：PRD中的设计方法和关注重点是否与产品类型匹配
  - 优质标准：
    - 业务型产品：重点关注业务流程、角色协同、数据模型、权限体系
    - 工具型产品：重点关注核心功能效率、操作体验、用完即走的便捷性
    - 交易型产品：重点关注交易流程闭环、支付清结算、商品管理、营销能力
    - 基础服务产品：重点关注接口设计、可扩展性、多业务线复用性、性能
  - 问题标志：设计重点与产品类型不匹配（如工具型产品却花大量篇幅设计复杂审批流程，或基础服务产品却重点设计业务逻辑）
  - 参考依据：《决胜B端》对四类产品特征的描述

- [ ] **单边市场与多边市场的识别**（仅交易型产品）
  - 检查内容：如果是交易型产品，是否明确了商业模型是单边市场还是多边市场（平台模式）
  - 优质标准：明确说明交易关系（企业直销/平台撮合/分销等），并据此设计了对应的角色体系（如多边市场需要设计商户入驻、商户管理功能）
  - 问题标志：交易型产品未说明商业模型，或平台型产品缺少供应商侧的设计
  - 参考依据：《决胜B端》1.3.3节交易型产品的单边市场与多边市场模式

### 2.2 部署模式与技术架构适配

- [ ] **部署模式明确**
  - 检查内容：PRD是否说明了产品的部署模式（私有化/云部署/混合部署）
  - 优质标准：说明部署模式选择的原因（如安全性要求、客户IT环境约束、成本考量），以及对产品设计的影响（如私有化部署需要考虑现场实施和运维）
  - 问题标志：未提及部署模式，或部署模式与客户需求不匹配（如面向银行客户却只考虑SaaS部署）
  - 参考依据：《决胜B端》1.2.2节——私有化部署"安全性强，稳定性好"，云部署"在保证安全性的前提下降低客户的运维成本"

- [ ] **技术架构类型**
  - 检查内容：是否说明了采用B/S还是C/S架构，以及端（PC Web/移动App/小程序/桌面客户端）的选择
  - 优质标准：根据用户使用场景选择合适的端和架构（如一线外勤人员需要移动端，办公室人员以PC Web为主），并说明选择理由
  - 问题标志：未考虑用户实际使用环境就确定了端的选择，或遗漏了关键端（如需要移动办公的业务只设计了PC端）
  - 参考依据：《决胜B端》1.2.2节B/S与C/S架构的区别

### 2.3 自研与商业化定位

- [ ] **产品运营模式定位**
  - 检查内容：PRD是否明确说明产品是企业自研自用还是商业化对外售卖
  - 优质标准：明确说明产品的商业属性，并据此调整设计策略——自研系统以满足内部需求为主、商业化产品需要考虑标准化和通用性
  - 问题标志：定位模糊，在自研和商业化之间摇摆（如"先内部用，以后可能卖给外部客户"而不考虑这对产品架构的影响）
  - 参考依据：《决胜B端》第2章明确区分了自研系统与商业化产品两条设计路径

- [ ] **标准化与个性化策略**（商业化产品必查）
  - 检查内容：如果是商业化产品（特别是SaaS），是否设计了标准化和个性化的平衡策略
  - 优质标准：明确标准功能范围和可配置/可定制的范围，有产品化策略（如通过配置中心、插件机制、开放API来满足个性化需求）
  - 问题标志：SaaS产品完全按照单一客户需求设计，无标准化考量；或过度标准化而忽视客户核心个性化诉求
  - 参考依据：《决胜B端》第8章产品化策略——标准化和个性化之间的平衡

### 2.4 B端产品核心特征合规

- [ ] **用户与客户的区分**
  - 检查内容：PRD是否区分了产品的用户（实际操作者）和客户（购买决策者）
  - 优质标准：分别定义了用户画像和客户画像，说明各自的核心诉求和关注点；对于SaaS产品，还需要考虑"试用者-购买者-使用者-管理者"等不同角色
  - 问题标志：用户和客户混为一谈，或只考虑了其中一方
  - 参考依据：《决胜B端》1.2.2节——"用户和客户是不同的人"是B端产品的核心特征之一

- [ ] **效能优先原则**
  - 检查内容：产品设计是否将业务效能放在首位，而非过度追求视觉体验
  - 优质标准：功能设计以解决业务问题为核心导向，在保证效能的基础上优化体验；不为了视觉美观牺牲操作效率
  - 问题标志：过度追求界面美观而忽视功能完整性；或者反过来，完全不考虑用户体验
  - 参考依据：《决胜B端》1.2.2节——"效能第一，体验第二"，同时《决胜体验设计》补充指出"不是不做体验，而是在效能基础上做好体验"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 工具型产品设计了复杂的审批流程 | 未识别产品类型，将工具型误当业务型 | 回归工具型产品核心：聚焦效率，场景单一，用完即走 |
| SaaS产品按单一客户需求定制 | 未理解商业化产品的标准化要求 | 区分标准功能和可配置功能，建立产品化策略 |
| 交易型产品缺少清结算设计 | 对交易型产品的完整能力闭环认知不足 | 补充OMS、支付、清结算、营销等交易核心能力 |
| 只有PC端设计，但用户主要在外勤 | 端选择未基于用户实际使用场景 | 基于用户画像和工作场景重新评估端选择 |
| 产品定位在自研和商业化之间摇摆 | 商业决策不明确，导致设计方向模糊 | 先明确商业定位，再据此确定标准化/定制化策略 |
