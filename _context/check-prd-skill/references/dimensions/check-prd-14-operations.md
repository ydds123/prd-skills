# 运营方案与效果跟踪检查

> Supporting reference for the `check-prd` skill. 在执行运营方案与效果跟踪检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品的成败不只取决于功能设计质量，更取决于能否落地运营、持续改善、真正被用户接受。《决胜B端》明确指出，产品效果受多因素影响：员工能力、业务流程合理性、管理制度——因此产品方案必须配套完整的运营方案才能形成闭环。

**核心理论框架：**

1. **产品运营体系五大类**（《决胜B端》产品运营体系设计）：
   - 需求收集与信息同步
   - 功能宣传与推广
   - 用户服务与支持
   - 上线跟踪与效果评估
   - 业务复盘与影响力构建

2. **产品监控指标体系**（《决胜B端》产品监控的指标体系设计）：覆盖用户数据、行为数据、业务数据、技术数据四类，自研系统和商业化产品都需要，差异仅在是否需要商业化指标。

3. **运营计划要求**（《决胜B端PRD模板》第13节）：需要新建的运营流程、业务流程、客服处理流程；需要执行的推广计划、部署计划、执行计划。

4. **数据驱动运营**（《决胜B端》第10章）：通过行为漏斗、页面转化率、用户留存等数据持续指导产品迭代，而非仅依赖主观判断。

## 检查清单

### 14.1 需求收集与信息同步

- [ ] **需求提报机制**
  - 检查内容：是否建立了统一的需求收集入口和周期性沟通机制
  - 优质标准：统一收口（避免多渠道散乱），定期需求沟通会（双周/月），有结构化的原始需求采集表，让业务方知道需求在哪个阶段
  - 问题标志：需求通过微信零散传达，无统一入口；业务方不知道自己提的需求是否会做、什么时候做
  - 参考依据：《决胜B端》产品运营体系设计——需求收集与信息同步

- [ ] **需求进展同步机制**
  - 检查内容：是否有持续向业务方同步需求进展的机制
  - 优质标准：信息同步表（日报表/周报表），定期沟通会，让业务方知道需求在哪个阶段
  - 问题标志：业务方反映"不知道我的需求做没做""不知道什么时候上线"
  - 参考依据：《决胜B端》产品运营体系设计

### 14.2 功能宣传与推广

- [ ] **产品发布节奏**
  - 检查内容：是否设计了固定的产品发布节奏和宣传方式
  - 优质标准：固定的发布周期（双周/月），让用户有预期；固定宣传方案（邮件/长图海报/短视频），根据受众选择合适渠道
  - 问题标志：功能上线后不通知用户，或仅发一封系统邮件；用户不知道新功能的存在和用法
  - 参考依据：《决胜B端》产品运营体系设计——功能宣传与推广

- [ ] **培训与推广落地**
  - 检查内容：新功能上线是否有配套的培训和使用激励方案
  - 优质标准：培训答疑（线上/线下），竞赛形式促进使用（技能大赛/知识大赛/最佳案例评选）
  - 问题标志：功能上线后用户不会用、不想用；缺少引导新用户了解功能价值的方案
  - 参考依据：《决胜B端》产品运营体系设计

### 14.3 用户服务与支持

- [ ] **问题反馈机制**
  - 检查内容：是否设计了用户反馈和问题响应的完整链路
  - 优质标准：明确受理入口（邮箱/系统意见箱），有问题受理流程和SLA（问题等级/响应速度/排班机制）
  - 问题标志：用户遇到问题不知道找谁；反馈了没有回音；问题处理无优先级无SLA
  - 参考依据：《决胜B端PRD模板》第13节——"客服处理流程"

- [ ] **意见激励机制**
  - 检查内容：是否有鼓励用户提意见并给予反馈的机制
  - 优质标准：意见收集入口，意见采纳后有反馈（物质激励/精神激励），让用户感受到被重视
  - 问题标志：用户提意见如石沉大海，久而久之不再反馈，形成信息孤岛
  - 参考依据：《决胜B端》产品运营体系设计

### 14.4 上线跟踪与效果评估

上线跟踪是产品持续改进的基础。自研系统和商业化产品都需要，差异在于是否还需要商业化指标。

- [ ] **跟踪责任与周期**
  - 检查内容：是否明确了谁负责跟踪、多长周期评估一次
  - 优质标准：明确跟踪负责人，有固定评估周期（周/月/季度），每个重要功能有专属的结果数据和使用数据追踪
  - 问题标志：产品上线后没有效果评估；无人负责跟踪
  - 参考依据：《决胜B端》产品监控的指标体系设计

- [ ] **用户数据与行为数据**
  - 检查内容：是否定义了观察用户状态和使用行为的指标
  - 优质标准：覆盖以下维度：
    - **用户数据**：用户类型（新用户/老用户），用户行为状态（活跃用户/流失用户/沉睡用户）
    - **行为数据**：活跃度（PV、UV、访问深度、访问时长），访问路径（行为漏斗、页面转化率），质量（弹出率、退出率、留存率）
  - 问题标志：只有结果指标（如销售额），没有行为数据；出了问题无法诊断原因
  - 参考依据：《决胜B端》产品监控的指标体系设计

- [ ] **业务数据与技术数据**
  - 检查内容：是否定义了衡量业务效果和系统健康的指标
  - 优质标准：覆盖以下维度：
    - **业务数据**：结果数据（GMV/销售额/转化率/满意度），过程数据（拜访量/通话时长/货损率等业务过程指标）
    - **技术数据**：接口调用（API调用次数），系统性能（并发量、流量、错误率）
  - 问题标志：指标全是结果指标，缺过程数据；技术健康无人监控
  - 参考依据：《决胜B端》产品监控的指标体系设计

### 14.5 业务复盘与影响力构建

- [ ] **定期复盘机制**
  - 检查内容：是否设计了定期业务复盘和拉通机制
  - 优质标准：定期复盘总结会（季度/半年/年度），业产研三方参与，优秀评选（优秀项目/优秀个人/最佳搭档），让参与者有成就感
  - 问题标志：只埋头做产品，缺少总结和展示；产品效果无人关注，改进缺乏方向
  - 参考依据：《决胜B端》产品运营体系设计——业务复盘与影响力构建

- [ ] **产研影响力构建**
  - 检查内容：是否有计划通过事件营销扩大产品和团队的影响力
  - 优质标准：通过事件营销扩大影响力（节日活动/周年纪念/黑客马拉松等），让产品团队在公司内部获得话语权
  - 问题标志：产品团队在公司内部缺乏话语权，投入资源做了产品却无人感知价值
  - 参考依据：《决胜B端》产品运营体系设计

### 14.6 上线准备：数据初始化与迁移

- [ ] **数据初始化与迁移方案**
  - 检查内容：新系统上线是否需要数据初始化或从旧系统迁移数据
  - 优质标准：定义了初始化数据的来源、格式、导入方式；如果有旧系统，定义了数据迁移方案（迁移范围、映射规则、数据清洗策略、迁移验证方法）
  - 问题标志：新系统需要历史数据但没有迁移方案，或上线后是空系统无初始数据，第一天打开是白板
  - 参考依据：B端产品上线准备工作

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 新系统上线没人用 | 缺少运营和培训方案 | 补充运营流程、培训计划、推广方案 |
| 业务方不知道需求进展 | 缺需求同步机制 | 建立固定的需求沟通会和进展同步表 |
| 上线后不知道效果好不好 | 缺指标体系和跟踪负责人 | 定义完整指标体系，指定专人负责跟踪 |
| 用户遇问题无处反馈 | 缺问题受理流程和SLA | 建立反馈入口，制定分级响应机制 |
| 产品团队话语权弱 | 缺成果展示和影响力建设 | 设计定期复盘机制，通过事件营销建立存在感 |
| 新系统第一天打开是空的 | 未提前规划数据初始化 | 在PRD中明确数据迁移和初始化方案 |
