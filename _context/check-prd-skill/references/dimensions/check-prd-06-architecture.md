# 架构设计质量检查

> Supporting reference for the `check-prd` skill. 在执行架构设计质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

架构设计是产品的骨架，决定了系统的扩展性、稳定性和与企业整体IT体系的融合程度。本维度分两层检查：

- **通用层（6.1-6.4）**：适用于所有 PRD，检查单系统维度的架构质量
- **企业架构层（6.5-6.9）**：仅适用于 **0-1 新系统设计或系统级规划 PRD**。如果 PRD 是迭代需求或模块级需求，6.5-6.9 标注"不适用"并跳过

**核心理论框架：**

1. **产品整体方案设计要素**（《决胜B端》第4章）：核心业务流程 → 产品定位 → 应用架构 → 功能模块与演进蓝图。应用架构是连接产品定位和功能细节的关键层。

2. **应用架构设计**（《决胜B端》4.2-4.3节）：包括系统间的关系（与公司现有产品体系如何融合）、系统内的功能模块划分、模块之间的依赖关系。

3. **企业应用版图规划**（《决胜B端》第16章）：将公司所有系统按业务能力域组织成一张地图，识别能力重叠、系统空白，作为新系统定位的依据。

4. **数据架构设计**（《决胜B端》产品规划部分 + 第16章）：明确企业数据域的划分，防止同一数据在多个系统中各自维护、互相矛盾。

5. **基础服务抽象**（《决胜B端》1.3.4节）：识别各系统共性需求（权限管理、消息推送、MDM、SSO等），抽象为基础服务，避免重复建设。

6. **功能架构与信息架构**（《决胜体验设计》第5章）：功能架构关注产品能力的结构化组织，信息架构关注内容和数据的组织方式。

## 【通用层】适用于所有 PRD

### 6.1 功能架构设计

- [ ] **功能架构图的层次性**
  - 检查内容：是否提供了产品的功能架构图，并体现层次结构
  - 优质标准：功能架构图从上到下分为 3-4 层（如：导航大类 → 功能模块 → 功能页面 → 核心功能点），层次清晰，一目了然
  - 问题标志：无功能架构图，或架构图只有一层（功能点平铺），看不出层次关系
  - 参考依据：《决胜体验设计》第5章功能架构设计

- [ ] **功能完整性与闭环**
  - 检查内容：功能设计是否构成完整的业务闭环，无"断头路"
  - 优质标准：从业务发起到业务结束，所有环节都有对应的功能支撑；数据在系统内流转无断点；每个操作都有明确的后续步骤
  - 问题标志：存在功能"断头路"——某个操作执行后，下一步无法继续（如创建了订单但没有发货功能，或审批通过后无后续处理流程）
  - 参考依据：《决胜B端》产品设计——"先做加法再做减法"，确保业务闭环后再做裁剪

- [ ] **功能菜单编排的合理性**
  - 检查内容：产品的功能菜单组织是否合理，用户是否能快速找到所需功能
  - 优质标准：菜单组织遵循业务逻辑或用户心智模型（如按业务阶段、按角色职能、按操作频率组织），层级不超过 3 层，高频功能在前
  - 问题标志：菜单按开发模块而非用户视角组织，层级过深（超过 4 层），高频功能隐藏在深层菜单中
  - 参考依据：《决胜体验设计》引用 Jesse 五要素之结构层——"让用户理解软件，找到所需"

### 6.2 系统边界与外部关系

- [ ] **系统架构图的存在与清晰度**
  - 检查内容：PRD 是否提供了系统架构图，展示产品在企业 IT 体系中的位置
  - 优质标准：架构图展示了本系统与周边系统的关系（上下游系统、依赖的基础服务、外部对接系统），标注了系统间的数据流向和交互方式（API/消息队列/文件等）
  - 问题标志：无架构图，或架构图只画了本系统内部结构而未考虑与外部系统的关系
  - 参考依据：《决胜B端》第4章——"规划产品的功能范围、定位，以及和公司现有产品体系如何融合"

- [ ] **基础服务的识别与复用**
  - 检查内容：是否识别了可以复用的基础服务（权限、消息、组织架构、SSO 等）
  - 优质标准：明确了哪些能力由公司现有基础服务提供，哪些需要自建，避免重复造轮子
  - 问题标志：每个功能模块都自建权限管理、消息推送等通用能力
  - 参考依据：《决胜B端》1.3.4节基础服务产品——MDM、Auth、Org、Push、SSO 等

- [ ] **系统集成与接口设计**
  - 检查内容：如果需要与外部系统集成，是否描述了接口设计
  - 优质标准：列出所有外部接口，每个接口说明了调用方向（主动/被动）、数据格式、频率、异常处理方式
  - 问题标志：提到了需要与外部系统集成但未设计接口，或接口描述不完整
  - 参考依据：《决胜B端》PRD 模板第6节项目范围——涉及哪些系统和接口

### 6.3 模块设计质量

- [ ] **功能模块划分的合理性**
  - 检查内容：系统内部的功能模块划分是否合理，模块之间的边界是否清晰
  - 优质标准：模块划分遵循高内聚低耦合原则；每个模块有明确的职责定义；模块之间的依赖关系清晰且无循环依赖
  - 问题标志：模块划分过细（导致模块间频繁交互）或过粗（一个模块承担过多职责）；模块边界模糊，同一功能在多个模块中重复
  - 参考依据：《决胜B端》应用架构设计原则

- [ ] **多端架构设计**
  - 检查内容：如果产品需要支持多端（PC Web、移动 App、小程序），是否有统一的多端架构设计
  - 优质标准：明确各端的功能差异和互补关系（如 PC 端做复杂操作，移动端做轻量级操作和审批），后端服务统一，前端按端特性差异化设计
  - 问题标志：多端设计各自为政，未考虑数据一致性和功能互补性
  - 参考依据：《决胜B端》PRD 模板 10.1.2 节功能清单按端标注

### 6.4 技术约束与演进

- [ ] **性能要求与约束**
  - 检查内容：PRD 是否描述了关键的性能要求（响应时间、并发量、数据量级）
  - 优质标准：标注了核心操作的响应时间要求，预估了用户并发规模和数据增长趋势
  - 问题标志：无任何性能要求说明，或要求不切实际
  - 参考依据：《决胜B端》第9章B端产品经理应具备的技术常识

- [ ] **架构的可扩展性与配置化**
  - 检查内容：架构设计是否为未来扩展预留了空间，业务规则是否设计为可配置
  - 优质标准：识别了可能的扩展方向，架构设计不会因扩展而需要大规模重构；将可变的业务规则（审批流、状态流转、字段校验规则）设计为可配置而非硬编码
  - 问题标志：架构紧密耦合，任何新增功能都需要修改核心模块；业务规则硬编码
  - 参考依据：《决胜B端》第8章产品化——通过配置中心、插件机制满足个性化需求

## 【企业架构层】仅适用于 0-1 新系统设计或系统级规划 PRD

> 如果 PRD 是迭代需求或模块级需求，以下 6.5-6.9 标注"不适用"并跳过。

### 6.5 企业应用版图定位

- [ ] **系统在企业 IT 版图中的位置**
  - 检查内容：PRD 是否说明了本系统在公司整体应用版图中的位置
  - 优质标准：说明了本系统所属的业务域（如"营销域"），与哪些现有系统存在功能重叠的边界，以及为什么选择新建而非扩展现有系统
  - 问题标志：新建系统与某个已有系统功能高度重叠但未说明区别；没有交代本系统与已有系统的关系
  - 参考依据：《决胜B端》第16章——"先理清家底，再决定建什么"

- [ ] **业务能力域划分**
  - 检查内容：是否明确了本系统负责哪些业务能力，不负责哪些
  - 优质标准：用业务能力（而非功能模块）描述系统边界，业务能力划分清晰，无模糊地带
  - 问题标志：系统边界描述含混，或一个系统试图覆盖多个能力域
  - 参考依据：《决胜B端》第16章业务能力地图方法论

- [ ] **make-or-buy 决策依据**
  - 检查内容：是否说明了为什么选择自研而非采购或复用
  - 优质标准：评估了市场上是否有成熟解决方案，说明了自研、采购 SaaS、定制化开发三种路径的选择逻辑
  - 问题标志：完全没有提及是否考虑过采购/复用；自研决策缺乏理由
  - 参考依据：《决胜B端》第16章企业IT建设的核心决策

### 6.6 数据架构设计

- [ ] **数据域划分与数据归属**
  - 检查内容：是否明确了本系统的数据边界——哪些数据由本系统负责管理（System of Record），哪些数据来自其他系统
  - 优质标准：列出了本系统的核心数据实体并标注数据主权；对于从外部系统获取的数据，说明了同步策略（实时读取 / 定期同步 / 本地副本）；避免同一数据在多个系统中各自维护
  - 问题标志：数据归属不清楚；同一业务实体（如"客户"）在多个系统中都有独立的完整数据集，存在一致性风险
  - 参考依据：《决胜B端》数据架构设计——"每份数据都应该有唯一的 System of Record"

- [ ] **主数据管理（MDM）策略**
  - 检查内容：如果涉及主数据（客户、产品、员工、组织等），是否说明了主数据的管理策略
  - 优质标准：明确了主数据的权威来源系统；说明了其他系统如何接入主数据（API 查询 / 数据同步）；有主数据变更时的通知机制
  - 问题标志：多个系统各自维护客户或产品数据，没有单一的主数据源
  - 参考依据：《决胜B端》1.3.4 节 MDM（主数据管理）作为基础服务的必要性

- [ ] **跨系统数据一致性**
  - 检查内容：跨系统的数据一致性问题是否被识别并设计了应对策略
  - 优质标准：识别了哪些数据需要在多个系统间保持一致，设计了最终一致性或强一致性方案；有数据对账和异常修复机制
  - 问题标志：存在明显的跨系统数据一致性需求但 PRD 未提及，或简单地说"通过接口同步"而无详细设计
  - 参考依据：《决胜B端》第16章数据架构治理

### 6.7 模块抽象与平台化决策

- [ ] **共享能力识别与抽象决策**
  - 检查内容：是否识别了本系统中哪些能力适合抽象为共享服务
  - 优质标准：识别了可被多个业务系统复用的通用能力（如规则引擎、审批流引擎、报表引擎）；对每个候选共享能力评估了抽象必要性（复用≥2次才值得抽象）；说明了抽象 vs. 内聚的决策依据
  - 问题标志：每个业务系统都各自实现审批流、消息推送等通用能力；或将不够通用的能力强行抽象为平台
  - 参考依据：《决胜B端》第16章模块抽象决策原则

- [ ] **平台化陷阱规避**
  - 检查内容：如果涉及平台化设计，是否规避了常见陷阱
  - 优质标准：平台的使用方明确，接入成本合理；有明确的平台治理机制（接入申请、版本管理、废弃策略）
  - 问题标志：平台设计过于抽象导致没有业务方愿意接入；或接入成本极高高于自建成本
  - 参考依据：《决胜B端》第16章平台化要避免的三大陷阱

### 6.8 集成治理

- [ ] **集成模式选择合理性**
  - 检查内容：与外部系统的集成采用了合适的集成模式
  - 优质标准：根据场景选择了合适的集成模式——实时 API（低延迟场景）、异步消息/事件驱动（松耦合最终一致）、批量 ETL（数据量大时效低）；不推荐直接跨库读取
  - 问题标志：所有集成都用实时 API 调用（高耦合、级联故障风险）；或业务系统直接读取另一个系统的数据库
  - 参考依据：《决胜B端》第16章集成架构设计

- [ ] **API 契约设计**
  - 检查内容：对外暴露的 API 是否有明确的契约设计
  - 优质标准：定义了 API 版本管理策略；说明了向后兼容原则和废弃策略；明确了哪些是稳定 API（对外承诺不变）哪些是内部 API
  - 问题标志：API 无版本管理，随时可能破坏性变更；对外部系统的依赖没有契约保障
  - 参考依据：《决胜B端》第16章 API 治理

### 6.9 技术治理一致性

- [ ] **技术选型与公司标准一致性**
  - 检查内容：本系统的技术选型是否与公司技术栈保持一致
  - 优质标准：在公司已有技术标准范围内做选择；如有偏差，有明确理由说明
  - 问题标志：引入了公司现有团队不熟悉的全新技术栈，增加学习成本和维护风险
  - 参考依据：《决胜B端》第16章技术治理——降低技术异质性

- [ ] **数据治理策略**
  - 检查内容：是否考虑了数据质量和数据治理问题
  - 优质标准：定义了核心数据实体的质量标准；有数据质量检测和修复机制；重要数据有数据血缘追踪；符合公司数据目录和元数据管理规范
  - 问题标志：数据质量无人负责；数据来源不清晰；数据出了问题无法追溯
  - 参考依据：《决胜B端》第16章数据治理框架

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 无系统架构图 | 直接进入功能设计，缺少宏观视角 | 先画应用架构图，明确系统边界和外部关系 |
| 功能有"断头路" | 未做端到端流程验证 | 沿核心业务流程走查，确保每个环节都有功能支撑 |
| 菜单层级过深用户找不到 | 按技术模块而非用户心智组织菜单 | 按业务流程或用户角色重新组织功能菜单 |
| 每个系统都自建通用能力 | 未识别和复用基础服务 | 梳理公司现有基础服务，能复用的不重复建设 |
| 新系统和老系统功能重叠 | 未做应用版图盘点 | 先梳理公司应用版图，明确新系统的差异化定位 |
| 同一数据在多个系统各自维护 | 未设计数据归属和 MDM 策略 | 明确每类数据的 System of Record，其他系统通过 API/订阅获取 |
| 系统间耦合严重，牵一发动全身 | 集成模式选择不当 | 引入事件驱动集成，减少直接依赖 |
| 平台建好没人用 | 平台抽象过度，接入成本高于自建 | 平台设计要以使用方为中心，降低接入成本 |
| 架构扩展困难 | 初始设计过度耦合，未预留扩展点 | 识别未来可能的扩展方向，在核心模块预留扩展接口 |
