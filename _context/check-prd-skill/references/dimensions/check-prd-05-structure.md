# 文档结构完整性检查

> Supporting reference for the `check-prd` skill. 在执行文档结构完整性检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

PRD文档的结构完整性直接影响评审效率和开发交付质量。一份结构清晰、章节完整的PRD能降低沟通成本、减少需求遗漏。《决胜B端PRD模板v2.0》提供了标准化的PRD文档结构，共包含14个章节。

**核心理论框架：**

1. **PRD标准文档结构**（《决胜B端PRD模板v2.0》）：项目背景 → 需求基本情况 → 商业分析 → 项目收益目标 → 项目方案概述 → 项目范围 → 项目风险 → 术语和缩略语 → 参考文献 → 功能需求（含产品框架概述、需求详解、异常处理） → 数据埋点 → 角色和权限 → 运营计划 → 待决事项。

2. **文档版本管理**（《决胜B端PRD模板v2.0》）：PRD交付前必须标记版本号，每次修改严格记录变更时间、原因、修改人、审核人。

3. **功能描述五种规则类型**（《决胜B端PRD模板v2.0》10.2节）：事实、约束、触发条件、推论、计算——功能规则描述必须归类到这五种类型之一，确保无遗漏。

4. **自顶向下的设计思路**（《决胜B端》第2章）：B端产品总体建设遵循"从抽象到具体"的原则，PRD文档结构应体现这一思路。

## 检查清单

### 5.1 文档元信息

- [ ] **文档头部信息完整性**
  - 检查内容：PRD是否包含审核人、重要性、紧迫性、需求方、编写人、提交日期等元信息
  - 优质标准：所有字段已填写，审核人和需求方明确到具体人名，日期格式统一
  - 问题标志：元信息缺失或填写不完整（如审核人写"待定"）
  - 参考依据：《决胜B端PRD模板v2.0》文档头部格式

- [ ] **版本变更记录**
  - 检查内容：是否维护了完整的PRD版本变更记录
  - 优质标准：每次修改都记录了变更时间、变更内容、变更提出部门与理由、修改人、审核人、版本号；版本号递增合理
  - 问题标志：无版本记录，或修改后未更新版本号，或变更记录不完整
  - 参考依据：《决胜B端PRD模板v2.0》明确指出"文档的版本管理是非常重要的工作，任何的文档修改都需要在文档中准确的记录"

### 5.2 前置章节完整性（第1-9节）

- [ ] **项目背景（第1节）**
  - 检查内容：是否有详细的项目背景描述
  - 优质标准：包含业务现状、面临问题、解决思路，有数据支持
  - 问题标志：背景空泛或缺失
  - 参考依据：PRD模板第1节——"不论是多简单的需求，都会有需求背景"

- [ ] **需求基本情况（第2节）**
  - 检查内容：是否包含需求提出人、功能使用人、受影响人、场景、发生频率、核心痛点、需求价值
  - 优质标准：七个字段全部填写，场景用六要素描述（人物、时间、地点、起因、经过、结果）
  - 问题标志：多个字段缺失，或场景描述过于简略
  - 参考依据：PRD模板第2节需求基本情况

- [ ] **商业分析（第3节）**
  - 检查内容：是否包含目标市场与客户分析、竞品分析两大板块
  - 优质标准：目标市场分析包含目标市场、市场规模、市场特征、发展趋势、客户画像、客户痛点、卖点提炼；竞品分析包含商业模式、目标客户、运营推广策略、市场份额、产品分析、核心价值
  - 问题标志：商业分析章节缺失或仅有空壳
  - 参考依据：PRD模板第3节商业分析表格

- [ ] **项目收益目标（第4节）**
  - 检查内容：是否有量化的、符合SMART原则的项目目标
  - 优质标准：目标具体可衡量，包含验收标准和成功标准，有收益预期
  - 问题标志：目标模糊或缺失
  - 参考依据：PRD模板第4节

- [ ] **项目方案概述（第5节）**
  - 检查内容：是否用简要语言概述了核心功能或项目方案
  - 优质标准：列举了所有核心功能feature，或概述了产品方案、运营方案、技术方案
  - 问题标志：无概述直接进入细节
  - 参考依据：PRD模板第5节

- [ ] **项目范围（第6节）**
  - 检查内容：是否明确了项目涉及的系统、产品、接口和影响范围
  - 优质标准：列出所有相关系统和责任方，标注影响范围，确保无遗漏
  - 问题标志：范围不明确，后续开发时才发现遗漏的关联方
  - 参考依据：PRD模板第6节——"梳理项目范围，提前确认项目关联方、影响方"

- [ ] **项目风险（第7节）**
  - 检查内容：是否识别并记录了假设、约束和各类风险
  - 优质标准：列出产品风险、运营风险、技术风险，每个风险都有应对方案
  - 问题标志：无风险分析，或只列风险不提应对方案
  - 参考依据：PRD模板第7节

- [ ] **术语和引用文档（第8-9节）**
  - 检查内容：是否解释了文档中的专业术语和缩略语，是否列出了参考文档
  - 优质标准：所有首次出现的术语有定义；引用的外部文档有明确链接
  - 问题标志：文档中充满缩略语但无解释，或引用了但找不到参考文档
  - 参考依据：PRD模板第8-9节——"尽量不要自己定义缩略语"

### 5.3 功能需求章节（第10节）

- [ ] **产品框架概述（10.1节）**
  - 检查内容：是否包含系统框架图、数据模型图、业务流程图、状态机图中的至少2-3种
  - 优质标准：四种图都有提供，框架图层次清晰，数据模型用ER图表达，流程图用跨部门泳道图表达
  - 问题标志：无任何框架概述图，直接进入页面级描述
  - 参考依据：PRD模板10.1节——"系统框架图、数据模型图、业务流程图、状态机图"

- [ ] **功能清单表格（10.1.2节）**
  - 检查内容：是否提供了子系统、页面、多端覆盖情况的功能清单
  - 优质标准：功能清单以表格形式呈现，包含子系统名称、页面名称、各端支持情况（PC/H5/App等），一目了然
  - 问题标志：无功能清单，或功能列举散落在文档各处
  - 参考依据：PRD模板10.1.2节功能清单表格格式

- [ ] **需求详解的规范性（10.2节）**
  - 检查内容：功能需求描述是否采用结构化方式（查询条件、列表字段、规则描述）
  - 优质标准：
    - 列表页有查询条件表格（字段名称、默认值、字段类型、备注）
    - 列表页有列表字段表格（字段名称、默认值、字段类型、开放修改、必输项、备注）
    - 规则描述按五种类型归类：事实、约束、触发条件、推论、计算
    - 每个功能模块有业务流程图和页面交互说明
  - 问题标志：功能描述为大段自然语言叙述，无结构化表格和分类
  - 参考依据：PRD模板10.2节的结构化描述格式和五种规则类型

- [ ] **异常处理方案（10.3节）**
  - 检查内容：是否有专门的异常情况处理方案章节
  - 优质标准：覆盖断网、断电、误操作、数据丢失、并发冲突等异常场景，每种异常有处理方案
  - 问题标志：无异常处理章节，或仅提"参见后续补充"
  - 参考依据：PRD模板10.3节

### 5.4 后置章节完整性（第11-14节）

- [ ] **数据埋点（第11节）**
  - 检查内容：是否按公司统一标准描述了需要监控的按钮、页面、事件
  - 优质标准：埋点设计覆盖核心操作路径，标注了事件名称、触发条件、采集参数
  - 问题标志：无埋点设计，或只有"后续补充"的占位符
  - 参考依据：PRD模板第11节

- [ ] **角色和权限（第12节）**
  - 检查内容：是否以权限矩阵的形式描述了各角色在各页面/功能上的权限
  - 优质标准：用表格形式列出一级导航、页面、页面元素与各角色的权限对应关系，精确到按钮级别
  - 问题标志：权限描述模糊（如"管理员有所有权限"），未精确到页面元素级别
  - 参考依据：PRD模板第12节角色权限矩阵格式

- [ ] **运营计划（第13节）**
  - 检查内容：是否说明了需要建立的运营流程、推广计划、部署计划
  - 优质标准：列出新建运营流程、客服处理流程、推广计划、部署方案
  - 问题标志：完全没有运营视角的考量
  - 参考依据：PRD模板第13节

- [ ] **待决事项（第14节）**
  - 检查内容：是否明确列出了文档中结论待定的事项
  - 优质标准：待决事项有编号、描述、负责人、预计决策时间，且数量不应过多（过多说明PRD成熟度不够）
  - 问题标志：明显有未决事项但未记录，或待决事项数量超过10项
  - 参考依据：PRD模板第14节

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| PRD只有功能描述，缺少背景和分析 | 把PRD当功能说明书写 | 按14节标准结构补充前9节 |
| 功能描述是大段文字 | 不熟悉结构化描述方法 | 采用表格+规则分类的标准格式 |
| 无版本记录 | 缺乏文档管理意识 | 建立版本管理规范，每次修改必记录 |
| 权限描述模糊 | 未做角色权限矩阵 | 按模板格式建立角色-页面-元素权限矩阵 |
| 待决事项过多 | PRD尚未成熟就匆忙评审 | 先解决核心待决事项再提交评审 |
| 缺少框架概述图 | 直接跳入细节设计 | 先完成系统框架图、ER图、流程图 |
