# 数据建模质量检查

> Supporting reference for the `check-prd` skill. 在执行数据建模质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

数据建模是B端产品设计中最核心、最具技术含量的工作之一。《决胜B端》第5章通过完整的对话案例（老马与小李的ER建模讨论），深入阐述了数据建模的方法论和思维过程。ER建模的质量直接决定了产品的灵活性和可扩展性——一旦数据库表结构定型，后续修改成本将指数级增加。

**核心理论框架：**

1. **ER建模方法论**（《决胜B端》5.1节）：通过ER图（Entity-Relationship图）描述业务实体和实体间的关系。ER建模是"帮你梳理业务的过程"，必须思考得"非常缜密、全面"。

2. **实体抽象原则**（《决胜B端》5.1节案例）：对业务对象进行合理抽象，避免冗余实体。案例中将"家长"和"孩子"抽象为"人"的维度，通过属性标记区分——体现了"先做加法再做减法"的建模思想。

3. **对应关系类型**（《决胜B端》6.3.1节）：一对一（1:1）、一对多（1:*）、多对多（*:*），每种关系都有特定的业务含义和实现方式。UML格式下用数字标注关系（如"0..*"表示0到多个）。

4. **数据模型的前瞻性**（《决胜B端》5.1节）：ER建模必须考虑未来可能的业务场景。"如果一上来就按照最简单的方案设计，未来想切换到理想模型，研发投入会非常巨大"。

5. **主数据管理MDM**（《决胜B端》1.3.4节）：企业在很多业务领域和主体中，应该有且只有一套相关的资料存储系统，保证数据管理的一致性和唯一性。

## 检查清单

### 7.1 ER图设计质量

- [ ] **ER图的存在与完整性**
  - 检查内容：PRD是否提供了核心业务实体的ER图
  - 优质标准：ER图覆盖了所有核心业务实体（如用户、组织、订单、商品、客户等），每个实体有关键属性说明，实体间关系用数字标注（1:1、1:*、*:*）
  - 问题标志：无ER图，或ER图只画了2-3个实体而业务涉及十几个核心对象
  - 参考依据：《决胜B端》PRD模板10.1.1节要求提供数据模型图

- [ ] **实体识别的完整性**
  - 检查内容：是否识别了所有核心业务实体，无重要实体遗漏
  - 优质标准：通过业务场景反推需要的实体——每个业务场景中的"主语"和"宾语"都应该有对应的实体；实体之间的关系能支撑所有已描述的业务场景
  - 问题标志：某些业务场景无法在数据模型上找到支撑（如PRD描述了"一个客户下有多个门店"，但ER图中客户和门店没有关联关系）
  - 参考依据：《决胜B端》5.1节——ER建模过程就是帮你梳理业务的过程

- [ ] **实体抽象的合理性**
  - 检查内容：实体的抽象层级是否合理，既不过度抽象也不过度具体
  - 优质标准：相似的业务对象有适当抽象（如将"家长"和"学生"抽象为"人"），但抽象不影响理解和实现；每个实体都有明确的业务含义
  - 问题标志：过度抽象（如将所有东西都抽象为"对象"，失去业务语义）；或不抽象（如家长、学生、教师分别建模，产生大量冗余实体和重复字段）
  - 参考依据：《决胜B端》5.1节老马与小李的对话——最终将家长和孩子抽象到"人"的维度

### 7.2 实体关系设计

- [ ] **关系类型的准确性**
  - 检查内容：实体间的对应关系类型是否准确反映了业务规则
  - 优质标准：
    - 一对一（1:1）：确实是业务约束（如一个账号对应一个员工）
    - 一对多（1:*）：方向和约束正确（如1个部门包含多个员工，而非反向）
    - 多对多（*:*）：有中间表来承载关系，中间表是否需要额外属性
  - 问题标志：关系类型标注不准确或缺失（如一对多画成了一对一），或多对多关系缺少中间实体
  - 参考依据：《决胜B端》6.3.1节ER图详解——实线两端标数字描述对应关系

- [ ] **关系的下限约束**
  - 检查内容：是否明确了关系的下限（0还是1）
  - 优质标准：区分了"0到多"（0..*，即可以没有）和"1到多"（1..*，即至少有一个）——这反映了不同的业务规则。例如"1个机构至少有1个门店"还是"可以没有门店"
  - 问题标志：所有一对多关系都标注为"1:*"而未区分下限是0还是1
  - 参考依据：《决胜B端》6.3.1节——ER图2"1..*的含义，至少要对应1个门店"

- [ ] **关系链的完整性**
  - 检查内容：关键实体之间的关系链是否完整，是否支撑核心业务场景
  - 优质标准：沿核心业务流程走查，确认每个环节涉及的实体都有正确的关系链连接。例如"客户→订单→商品→发货单→物流单"的完整链路
  - 问题标志：关系链存在断裂——两个在业务上相关的实体在ER图中没有直接或间接关系
  - 参考依据：《决胜B端》5.1节——ER模型必须支撑各种业务诉求

### 7.3 核心实体详细设计

- [ ] **实体属性的完整性**
  - 检查内容：核心实体是否列出了关键属性（字段）
  - 优质标准：至少列出实体的标识字段（ID）、核心业务字段、状态字段、时间戳字段（创建时间/更新时间），关键字段说明了数据类型和约束条件
  - 问题标志：实体只有名称没有属性，或关键业务字段遗漏
  - 参考依据：《决胜B端》6.3.1节ER图格式——方框第二行描述对象中的数据字段

- [ ] **状态设计的完整性**
  - 检查内容：有状态变化的实体（如订单、审批单、工单）是否有完整的状态机设计
  - 优质标准：提供了状态机图（State Machine Diagram），列出所有状态及状态间的转换条件；状态命名清晰无歧义；无"死状态"（无法流转到其他状态的死胡同）
  - 问题标志：有状态实体但无状态机图，或状态流转不完整（如"已提交"后缺少审批的状态变化路径）
  - 参考依据：《决胜B端》6.3节UML图——状态机图是产品经理常用的UML图之一

- [ ] **字段级别的规则描述**
  - 检查内容：PRD是否按字段描述了数据规则（默认值、校验规则、联动规则）
  - 优质标准：列表页和表单页的字段有完整的规则描述——字段名称、默认值、字段类型、是否必输、开放修改、长度限制、校验规则等
  - 问题标志：字段描述缺失，或只有字段名称而无规则
  - 参考依据：《决胜B端》PRD模板10.2节——查询条件和列表字段的结构化表格格式

### 7.4 数据治理考量

- [ ] **数据一致性保障**
  - 检查内容：是否考虑了跨模块、跨系统的数据一致性
  - 优质标准：识别了数据主源（谁是数据的Owner），定义了数据同步策略，说明了数据冲突的处理方式
  - 问题标志：同一数据在多处维护但未说明如何保持一致
  - 参考依据：《决胜B端》1.3.4节MDM主数据管理——"有且只有一套相关的资料存储系统"

- [ ] **数据模型的前瞻性**
  - 检查内容：数据模型是否考虑了未来可能的业务变化
  - 优质标准：关键实体的设计预留了扩展空间（如用可扩展的属性字段而非硬编码），对可能的业务演变（如从单一客户类型到多客户类型）有预判
  - 问题标志：数据模型只满足当前需求，一旦业务变化就需要大规模重构
  - 参考依据：《决胜B端》5.1节——"如果一上来就按最简单的方案设计，未来想切换到理想模型，研发投入会非常巨大"

- [ ] **历史数据和数据归档**
  - 检查内容：是否考虑了历史数据的处理和数据归档策略
  - 优质标准：说明了数据保留周期、归档策略、历史数据查询需求；对于数据量大的实体，有分表/分库的预案
  - 问题标志：未考虑数据增长带来的问题，或需要保留的历史数据会被覆盖或删除
  - 参考依据：《决胜B端》数据建模最佳实践

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 无ER图 | 跳过建模直接画页面 | 先画ER图，再设计页面和交互 |
| 实体过多过碎 | 未做合理抽象 | 识别相似实体进行抽象合并 |
| 实体过于抽象 | 过度工程化设计 | 确保抽象不影响业务语义理解 |
| 关系类型错误 | 业务理解不够深入 | 用具体业务场景验证每个关系 |
| 无状态机图 | 遗漏状态管理设计 | 对所有有状态的实体补充状态机图 |
| 数据模型改一次架构崩一次 | 初始建模不够前瞻 | 先做加法考虑所有场景，再做减法选择合适方案 |
| 同一数据多处维护 | 缺乏主数据管理意识 | 确定数据主源，建立数据同步机制 |
