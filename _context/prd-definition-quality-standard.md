# PRD 定义与质量标准

> 建议路径：`05_context/prd-standards/good-prd-definition.md`  
> 适用对象：`prd-thinking`、`prd-writer`、`prd-reviewer`、`prd-reviser`、`skill-retrospector`、各类 PRD 质量门禁  
> 版本：v0.2.0  
> 状态：可作为 PRD Skill Stack 的质量标准草案使用

---

## 1. 核心结论

一份好的 PRD，不是把功能列全，而是把产品判断固定成团队可执行、可验证、可追责的共同答案。

它的唯一质量标准是：

> 读的人不需要再回来找 PM 确认。

这条标准继续拆成四个核心指标：

| 核心指标 | 一句话定义 | 解决的问题 |
|---|---|---|
| 边界清晰 | 本期做什么、不做什么、适用于谁、不适用于谁，都写清楚 | 防止范围蔓延和理解跑偏 |
| 判断显性 | 关键取舍、规则、异常、依赖、责任方都写出结论和依据 | 防止团队各自脑补 |
| 不留猜疑 | 默认值、触发条件、输入输出、状态变化、验收方式都可直接执行 | 防止研发、测试、设计继续追问 |
| 信息准确 | 已确认事实、假设、待确认项、数据口径和上下文来源清楚区分 | 防止把不确定内容写成确定事实 |

后续所有 PRD 生成、审核、修订和 Skill 迭代，都应围绕这四个指标展开。

---

## 2. 标准来源说明

本文档包含两层标准。

| 标准层级 | 来源 | 用途 |
|---|---|---|
| 课件原始标准 | `pm-ai-course.md` 中关于 PRD 本质、质量标准、三项指标和 AI 实现新尺子的内容 | 定义“好 PRD”的根标准 |
| 本项目工程化扩展 | 为支持 PRD Skill 生成、审核、门禁和自优化而补充的评分、阻塞、检查清单 | 让标准可以被 Skill 稳定执行 |

使用原则：

1. 课件原始标准优先于工程化扩展。
2. 工程化扩展只能帮助执行原始标准，不能反向改写原始标准。
3. 当评分模型、门禁规则与核心定义冲突时，以核心定义为准。
4. 当文档写得很完整，但读的人仍需回来确认时，仍判定为质量不达标。

## 3. PRD 的定义

### 3.1 PRD 是什么

PRD 是把产品经理脑中的判断固定下来，让团队在同一个需求上形成同一个答案。

它要回答的不是“系统支持什么功能”，而是：

| 问题 | PRD 中应给出的答案 |
|---|---|
| 为什么做 | 解决谁的什么问题，为什么现在要做 |
| 做到哪 | 本期目标、范围、边界和非目标是什么 |
| 怎么判断 | 关键规则、默认值、异常处理、依赖和验收标准是什么 |
| 谁参与 | 使用者、配置者、审批者、维护者、受影响者是谁 |
| 怎么交付 | 研发、测试、设计、运营分别能依据 PRD 做什么 |

### 3.2 PRD 不是什么

| PRD 不应该是 | 原因 |
|---|---|
| 功能清单 | 只列功能，不写判断，容易做出“能跑但没用”的东西 |
| 会议纪要 | 会议纪要记录讨论，PRD 固定最终判断 |
| 技术方案 | 技术实现属于研发判断空间，PRD 只定义产品行为 |
| 视觉规范 | 色号、像素、字体等属于设计判断空间，PRD 只定义操作优先级和状态要求 |
| 愿望清单 | “提升体验”“优化流程”不是可执行需求 |
| AI 一次性生成物 | PRD 必须经过写前对齐、人工确认和质量门禁 |

## 4. 好 PRD 的唯一质量标准

一份高质量 PRD 的标志是：读的人不需要再回来找 PM 确认。

这里的“读的人”至少包括：

| 角色 | 读完后应该获得什么 |
|---|---|
| 研发 | 知道做什么、边界是什么、异常怎么处理、如何判断开发完成 |
| 测试 | 知道主流程、分支流程、异常流程、验收标准和测试重点 |
| 设计 | 知道页面目标、信息优先级、关键状态和操作约束 |
| 运营 / 业务 | 知道需要配置什么、维护什么、关注什么结果 |
| PM 自己 | 三个月后仍能看懂当时为什么这样定，而不是重新猜一遍 |

如果读者读完后仍频繁追问以下问题，说明 PRD 没有达标：

| 追问类型 | 暴露的问题 |
|---|---|
| 这个需求到底解决什么问题？ | 问题和目标不清 |
| 这个本期做不做？ | 范围和非目标不清 |
| 这个场景怎么处理？ | 规则、异常或边界不清 |
| 这个数据从哪里来？ | 数据来源和口径不清 |
| 这个角色能不能操作？ | 权限和角色不清 |
| 做完怎么验收？ | 验收标准不清 |
| 这里是不是你默认的意思？ | 仍然存在脑补空间 |

## 5. 四个核心质量指标

### 5.1 边界清晰

边界清晰，指 PRD 必须明确本期需求的适用范围和不适用范围。

| 检查项 | 必须写清的问题 | 不达标信号 |
|---|---|---|
| 功能边界 | 本期做什么、不做什么 | 只写“支持某功能”，没写哪些不支持 |
| 用户边界 | 哪些角色使用，哪些角色不使用 | 所有人都能用，权限默认混在一起 |
| 场景边界 | 哪些业务场景覆盖，哪些场景暂不覆盖 | 只写正常场景，不写边缘场景 |
| 数据边界 | 使用哪些数据，不使用哪些数据 | 数据来源、口径、更新频率不明 |
| 版本边界 | 本期、后续、暂不做分别是什么 | 把后续规划混进本期需求 |
| 责任边界 | 谁配置、谁维护、谁审批、谁处理异常 | 责任人留给团队猜 |

最低要求：

1. 必须有“本期做什么”。
2. 必须有“本期不做什么”。
3. 必须说明适用角色和适用场景。
4. 对可能被误解为本期范围的内容，必须明确排除。

---

### 5.2 判断显性

判断显性，指 PRD 不能只写“要什么”，还要写清楚为什么这样定、按什么规则执行、出了岔子怎么办。

| 检查项 | 必须写清的问题 | 不达标信号 |
|---|---|---|
| 问题判断 | 用户真实问题是什么 | 只复述用户提出的方案 |
| 目标判断 | 本期做到什么程度算有效 | 只有“优化”“提升”“完善”等空话 |
| 方案取舍 | 为什么选 A，不选 B | 没有说明被排除方案 |
| 业务规则 | 什么条件下允许 / 禁止 / 触发 | 只写“支持配置”“支持审批” |
| 默认规则 | 用户不操作时系统默认怎么处理 | 默认值留给研发决定 |
| 异常判断 | 空态、失败、冲突、重复、撤回怎么处理 | 只写正向流程 |
| 依赖判断 | 依赖哪些上游和下游 | 上下游影响不明 |
| 责任判断 | 谁负责配置、维护、处理 | “由相关人员处理”这类模糊表述 |

最低要求：

1. 每个核心功能都必须写规则。
2. 每个核心功能都必须写异常或说明无异常。
3. 关键默认值必须显式说明。
4. 有分歧的选择题必须进入决策账本。

---

### 5.3 不留猜疑

不留猜疑，指 PRD 必须把判断翻译成可执行说明，让读者无需脑补。

| 检查项 | 必须写清的问题 | 不达标信号 |
|---|---|---|
| 触发条件 | 什么情况下进入这个功能或流程 | “当需要时”“适当时” |
| 操作入口 | 从哪里进入，谁能进入 | 入口位置和权限不明 |
| 输入输出 | 用户输入什么，系统输出什么 | 只写操作，不写结果 |
| 状态变化 | 操作前、操作中、操作后状态如何变化 | 状态流转断裂 |
| 结果反馈 | 成功、失败、处理中如何提示 | 只写提交，不写反馈 |
| 验收方式 | 怎么证明功能完成且正确 | “功能正常即可” |
| 自测用例 | 研发交付前至少跑哪些主流程 | 验收时才发现主流程不通 |

最低要求：

1. 核心流程必须能从入口走到结束。
2. 核心状态必须能闭环。
3. 验收标准必须可验证。
4. 不使用“适当、合理、友好、智能、优化、完善”等无法执行的空词，除非后面给出明确判定标准。

---

### 5.4 信息准确

信息准确，指 PRD 中的事实、数据、假设、待确认项必须分清楚，并与已有上下文保持一致。

| 检查项 | 必须写清的问题 | 不达标信号 |
|---|---|---|
| 事实准确 | 已确认信息是否与上下文一致 | 与已有 PRD、业务规则、系统现状冲突 |
| 数据准确 | 数据来源、口径、时间范围是否明确 | “约 30%”但没有来源 |
| 状态准确 | 已确认、待确认、默认采用是否区分 | 把待确认内容写成确定事实 |
| 术语准确 | 同一个对象是否使用同一名称 | 模板、方案、配置混用 |
| 依赖准确 | 上下游是否真实存在 | 写了不存在的系统能力 |
| 能力准确 | 不夸大系统或 AI 当前能力 | 把“未来可做”写成本期已支持 |

最低要求：

1. 未确认的数据必须标注“待确认”。
2. 未确认的规则必须进入“待确认事项”或“决策账本”。
3. 不得编造系统能力、数据口径、用户比例或外部依赖。
4. 与已有上下文冲突时，必须指出冲突并要求确认，不得自行选择。

## 6. 两个高风险必写项

以下两项是高质量 PRD 的重点，也是新手最容易漏、最容易导致返工的部分。

### 6.1 本期不做什么

“本期不做什么”用于防止范围蔓延。

| 必写内容 | 示例写法 |
|---|---|
| 明确排除的功能 | 本期不做智能问答，仅支持关键词别名匹配 |
| 明确排除的角色 | 本期仅支持运营配置，暂不开放给普通用户配置 |
| 明确排除的数据 | 本期不接入实时日志，仅使用 T+1 统计数据 |
| 明确排除的场景 | 本期不覆盖历史数据迁移，仅对新产生的数据生效 |
| 后续规划但本期不做 | 语义搜索作为后续规划，不进入本期验收范围 |

判断标准：

1. 如果某项内容容易被研发、测试、业务误认为本期要做，就必须写入“不做什么”。
2. 如果某项内容会明显影响工作量，就必须明确是否本期覆盖。
3. 如果某项内容只是未来想法，不得混入本期范围。

---

### 6.2 异常情况

“异常情况”用于定义出岔子时系统和人应该怎么处理。

| 异常类型 | 必须说明的问题 |
|---|---|
| 空态 | 没有数据时展示什么，用户能做什么 |
| 失败态 | 提交失败、保存失败、加载失败怎么处理 |
| 重复态 | 重复提交、重复选择、重复配置怎么处理 |
| 冲突态 | 数据被别人修改、状态已变化时怎么处理 |
| 权限不足 | 用户无权限时是否展示入口，如何提示 |
| 状态不可逆 | 删除、撤回、关闭、终止等操作是否允许恢复 |
| 上下游异常 | 依赖系统无响应、数据未同步时怎么兜底 |

判断标准：

1. 每个核心功能都必须至少说明一种异常或明确“无特殊异常”。
2. B 端流程类需求必须写状态异常和权限异常。
3. 数据类需求必须写数据为空、数据延迟、数据口径不一致的处理方式。
4. AI 功能类需求必须写低置信度、无结果、错误结果和人工介入机制。

## 7. AI 实现视角的新尺子

当 PRD 会被 AI 用于辅助实现时，还需要增加一把新尺子：

> 把 PRD 交给 AI 实现，最终实现结果和预期差多少；差距越大，说明 PRD 中漏写、写虚或写错的地方越多。

AI 实现者不会自动替 PM 补判断。PRD 中没有写清的内容，AI 很可能会按字面理解、默认补全或直接遗漏。

| AI 容易出错的位置 | PRD 应补充的内容 |
|---|---|
| 默认值没写 | 明确默认状态、默认排序、默认筛选、默认开关 |
| 异常没写 | 明确空态、失败态、冲突态、权限不足处理 |
| 边界没写 | 明确哪些角色、数据、场景不覆盖 |
| 依赖没写 | 明确上游来源、下游影响、同步时机 |
| 验收没写 | 明确可验证用例，而不是只写功能描述 |
| 假设没标 | 标明“待确认”，避免 AI 当成事实实现 |

AI 实现就绪检查：

| 检查项 | 通过标准 |
|---|---|
| 无隐含默认值 | 所有关键默认值已写明 |
| 无未标记假设 | 假设、待确认、默认采用均有标记 |
| 输入输出清楚 | 每个核心功能都有输入、处理、输出 |
| 异常兜底清楚 | 失败、空态、冲突、权限等有处理 |
| 验收用例明确 | 可以据此生成测试用例或研发自测用例 |

## 8. 工程化质量维度

> 来源类型：本项目工程化扩展。  
> 说明：本节用于 PRD Skill 的生成、审核和质量门禁，不是对课件原始标准的替代。

为了让 Skill 可执行，四个核心指标可展开为 9 个质量维度。

| 质量维度 | 分值 | 对应核心指标 | 检查重点 |
|---|---:|---|---|
| 问题与价值 | 12 | 判断显性、信息准确 | 是否说清真实问题，而非照搬用户方案 |
| 目标与成功标准 | 8 | 判断显性、不留猜疑 | 是否能判断做完是否有效 |
| 范围与非目标 | 15 | 边界清晰 | 是否写清本期做什么、不做什么 |
| 角色与场景 | 10 | 边界清晰、不留猜疑 | 是否说明谁在什么场景下使用 |
| 流程与状态 | 12 | 不留猜疑 | 主流程、分支、逆向、状态变化是否闭环 |
| 功能规则与关键决策 | 13 | 判断显性 | 功能规则、默认值、责任方、依赖是否清楚 |
| 异常与兜底 | 10 | 判断显性、不留猜疑 | 空态、失败、冲突、权限、重复是否覆盖 |
| 验收与自测 | 10 | 不留猜疑 | 是否可验证，研发能否交付前自测 |
| 信息准确与边界不过界 | 10 | 信息准确 | 是否事实准确、术语一致、不写技术实现和设计参数 |
| 合计 | 100 |  |  |

评分解释：

| 分数 | 判断 | 处理建议 |
|---:|---|---|
| 90-100 | 质量较高 | 可进入需求评审 |
| 80-89 | 基本可用 | 修正少量 P2 后进入评审 |
| 70-79 | 有明显缺口 | 不建议直接评审，需修订 |
| 60-69 | 返工风险高 | 回到修订或写前对齐 |
| 60 以下 | 不合格 | 重新对齐需求并重写 |

注意：评分不能替代门禁。即使总分较高，只要存在 P0 或未处理 P1，仍不得判定为可评审。

## 9. 阻塞问题标准

> 来源类型：本项目工程化扩展。  
> 说明：本节用于 `prd-reviewer`、`review-blocking-gate` 和 `final-output-gate`。

### 9.1 P0：致命问题，必须阻塞

| P0 问题 | 判断标准 |
|---|---|
| 真实问题不清 | 不知道解决谁的什么问题 |
| 本期范围不清 | 做什么、不做什么无法判断 |
| 核心流程不可执行 | 主流程无法从入口走到结束 |
| 关键决策缺失 | 默认值、责任方、权限、核心规则未定 |
| 未确认事实被写成确定事实 | 数据、结论、系统能力没有来源却写成事实 |
| 与已有上下文冲突 | 与现有 PRD、业务规则、系统现状明显冲突 |
| 明显越界 | 用 PRD 直接规定技术实现或视觉参数，且影响研发 / 设计判断空间 |

处理规则：

1. 存在任何 P0，不得输出最终版 PRD。
2. P0 必须修正，不能仅标记风险接受。
3. 修正 P0 后必须重新审核相关章节。

---

### 9.2 P1：高风险问题，默认阻塞

| P1 问题 | 判断标准 |
|---|---|
| 缺少“不做什么” | 存在明显范围误解风险 |
| 缺少异常处理 | 核心功能只有正常流程 |
| 缺少权限规则 | 多角色场景下未说明谁能做什么 |
| 缺少状态变化 | 审批、流转、启停、关闭等状态不清 |
| 缺少验收标准 | 做完后无法判断是否完成 |
| 缺少上下游影响 | 依赖和影响对象不明 |
| 缺少数据口径 | 数据来源、统计口径、更新时间不明 |

处理规则：

1. P1 默认阻塞最终输出。
2. 如 PM 明确接受风险，必须在 PRD 中记录“已接受风险”和原因。
3. 未被接受的 P1 必须修正后才能进入最终版。

---

### 9.3 P2：质量问题，建议修改

| P2 问题 | 判断标准 |
|---|---|
| 表达不够清楚 | 有歧义，但不影响核心判断 |
| 示例不足 | 规则有说明，但缺少例子帮助理解 |
| 验收不够细 | 能验收，但测试成本偏高 |
| 章节组织不佳 | 信息完整，但阅读路径不顺 |
| 局部术语不一致 | 不影响主流程，但会增加理解成本 |

处理规则：

1. P2 不默认阻塞最终输出。
2. 进入评审前建议尽量修正。
3. 多个同类 P2 反复出现时，应进入 Skill 迭代清单。

---

### 9.4 P3：优化建议，不阻塞

| P3 问题 | 判断标准 |
|---|---|
| 可读性优化 | 更换表达方式会更清楚 |
| 结构优化 | 调整顺序会更易读 |
| 示例增强 | 增加示例能提升理解 |
| 格式统一 | 表格、编号、命名可以更整齐 |

处理规则：

1. P3 不阻塞最终输出。
2. 仅在不影响交付节奏时处理。
3. 不得因为 P3 反复扩大 PRD 范围。

## 10. 复杂度适配原则

> 来源类型：本项目工程化扩展。  
> 说明：本节用于避免小需求被重型 PRD 模板拖慢，也避免复杂需求写得过轻。

| 等级 | 需求类型 | PRD 要求 |
|---|---|---|
| L1 | 文案、字段、配置、小规则调整 | 可用变更说明，必须写清变更点、影响范围、验收方式 |
| L2 | 单页面、单流程功能 | 标准 PRD，必须覆盖背景、范围、流程、规则、异常、验收 |
| L3 | 多角色、多状态、多异常功能 | 完整 PRD，必须覆盖角色权限、状态流转、异常、上下游 |
| L4 | 跨系统、跨模块、数据 / 权限 / AI 能力类需求 | 完整 PRD + 风险评审 + AI 实现就绪检查 |

适配原则：

1. 复杂度可以影响文档篇幅，不能降低核心判断质量。
2. L1 可以简化章节，但不能省略范围、规则和验收。
3. L3/L4 必须重点检查权限、状态、异常、数据和上下游影响。
4. AI 功能类需求无论大小，都必须写兜底、置信度、人工介入和错误处理。

## 11. PRD 应写与不应写

### 11.1 PRD 应写

| 应写内容 | 说明 |
|---|---|
| 背景与问题 | 为什么做，解决谁的问题 |
| 本期目标 | 做到什么程度算有效 |
| 范围与非目标 | 本期做什么、不做什么 |
| 用户与角色 | 谁使用、谁配置、谁审批、谁受影响 |
| 业务流程 | 主流程、分支流程、逆向流程 |
| 功能规则 | 触发条件、操作规则、默认值、结果反馈 |
| 异常处理 | 空态、失败、冲突、重复、权限不足 |
| 权限规则 | 不同角色能看什么、做什么、不能做什么 |
| 数据说明 | 数据来源、字段含义、统计口径、更新时间 |
| 验收标准 | 如何判断完成 |
| 研发自测用例 | 交付前能快速验证主流程是否可用 |
| 待确认事项 | 尚未确认但会影响实现或评审的问题 |

### 11.2 PRD 不应写

| 不应写内容 | 替代写法 |
|---|---|
| 具体技术实现 | 写产品行为和业务规则 |
| 数据库表结构 | 写需要记录哪些业务信息和口径 |
| 接口字段细节 | 写输入、输出和业务含义 |
| 代码逻辑 | 写规则和结果 |
| 色号、像素、字号 | 写操作优先级、状态要求、提示强度 |
| 营销式空话 | 写具体目标和可验证标准 |
| 未确认事实 | 标注“待确认” |
| 未来想法混入本期 | 放入“后续规划”或“本期不做” |

## 12. 最低合格清单

在进入需求评审前，PRD 至少要满足以下清单。

| 检查项 | 是否必过 | 对应核心指标 |
|---|---|---|
| 写清真实问题 | 是 | 判断显性、信息准确 |
| 写清本期目标 | 是 | 判断显性 |
| 写清本期做什么 | 是 | 边界清晰 |
| 写清本期不做什么 | 是 | 边界清晰 |
| 写清核心流程 | 是 | 不留猜疑 |
| 写清核心功能规则 | 是 | 判断显性 |
| 写清异常和空态 | 是 | 判断显性、不留猜疑 |
| 写清角色和权限 | B 端需求必过 | 边界清晰、不留猜疑 |
| 写清验收标准 | 是 | 不留猜疑 |
| 标记待确认项 | 是 | 信息准确 |
| 不写技术实现 | 是 | 信息准确 |
| 不写设计参数 | 是 | 信息准确 |

若任一“必过”项缺失，PRD 不应标记为最终版。

## 14. 最终判断公式

可以用下面的公式判断一份 PRD 是否合格：

```text
好 PRD = 固定判断 + 清晰边界 + 显性规则 + 完整异常 + 可验证验收 + 准确信息
```

也可以用下面的问题做最终检查：

```text
研发读完能不能做？
测试读完能不能测？
设计读完能不能画？
业务读完能不能确认？
PM 三个月后能不能看懂当时为什么这么定？
AI 拿去实现时，会不会因为漏写一句话而做偏？
```

如果以上问题有明显否定答案，这份 PRD 就还没有达到最终输出标准。

## 15. 版本维护规则

本文档后续可以更新，但必须遵守以下规则：

1. 不得删除“PRD 是固定判断”这一核心定义。
2. 不得删除“读的人不需要再回来确认”这一唯一质量标准。
3. 不得删除“边界清晰、判断显性、不留猜疑、信息准确”四个核心指标。
4. 工程化扩展可以调整，但必须标注来源类型和适用范围。
5. 新增评分项或门禁项时，必须说明它对应哪个核心指标。
6. 因一次偶发错误产生的规则，不应直接写入本文档；需进入 Skill 迭代清单，经确认后再采纳。
