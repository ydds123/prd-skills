# Run Log Template

> Belongs to: `05_context/`  
> Purpose: 跨节点的持续性运行日志 — 决策追加、修订根因、痛点证据  
> Governed by: `workflow-protocol.md` §Run Log Checkpoints  
> Version: v1.0.0

---

## 为什么需要 Run Log

PRD workflow 的五个节点分布在多次对话中。复盘时 Agent 只能靠对话记忆推导"为什么初版漏了交互逻辑"——但对话记忆随上下文长度衰减。Run Log 是这些证据的持久载体。

它不替代任何现有文件：
- `02-决策账本.md` 是对齐阶段的决策快照，Run Log 记录对齐后**新增的**决策
- `07-修订记录.md` 记录"改了什么"，Run Log 记录"**为什么漏了、第几轮发现的、根因是什么**"
- `08-复盘建议.md` 是最终产物，Run Log 是**生成它所需的首要证据**

## 写入时机

| 时机 | 写入字段 | 触发规则 |
|------|---------|---------|
| 每次节点开始 | 运行时间线 | 新节点启动时追加一行 |
| 每次节点完成 | Node 完成记录 | 更新该节点的结束时间和统计 |
| 对齐后出现新决策 | 决策追加 | 即时追加（不回写 02-决策账本） |
| 每次修订完成 | 修订记录 | 即时追加，必须填"根因分类"和"checklist 模块" |
| 发现"初版不该漏"的问题 | 痛点日志 | 即时追加，必须填"初版为什么漏了" |

## 复盘消费规则

复盘 Agent 启动时：

1. 先读 Run Log 的"修订记录"和"痛点日志"作为首要证据
2. 统计"根因分类"分布：缺方法 X 次 / 缺模板 Y 次 / 缺门禁 Z 次
3. 同类根因 ≥ 2 次 → 必须出 patch
4. 同类根因 ≥ 3 次 → patch 优先级升至 P0
5. 偶发（1 次）→ 仅记录，不出 patch

复盘完成后，在 Run Log 末尾追加：

```markdown
## 复盘消费

| 时间 | 消费的痛点 | 产出的 patch | 是否已写入 |
|------|-----------|-------------|-----------|
| | | | 是 / 否 |
```

---

## 禁止行为

- 不得在 Run Log 中重复写入 02-决策账本已有的内容
- 不得跳过"根因分类"字段 — 没有分类的修订记录对复盘无用
- 不得在复盘消费前删除 Run Log
- 不得等"整个 PRD 做完"再一次性补写 Run Log — 它是实时日志，不是事后回忆
