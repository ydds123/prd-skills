# 风险接受表

> Belongs to: `05_context/writing-standards/`  
> Source: PRD 需求自查清单 V3.3 · Sheet 07  
> Version: v1.0.0

适用场景：P1 风险被暂时接受、待确认项不阻塞、历史数据暂不处理、异常场景延期。

风险接受不是把问题藏起来，而是写清：谁接受，为什么接受，影响谁，什么时候补，不补会怎样。

关键提醒：P1 默认阻塞，除非 PM 明确接受风险并记录原因。

## 空白模板

```
风险编号 | 风险类型 | 未解决问题 | 影响范围 | 问题等级 | 接受原因 | 接受人/责任人 | 后续处理 | 截止时间 | 记录日期
```

| 列名 | 含义 |
|------|------|
| 风险编号 | 唯一标识，如 RA001 |
| 风险类型 | 数据口径 / 历史数据 / 异常处理 / 权限边界 / 性能边界 / 外部依赖 / 待确认 |
| 未解决问题 | 具体是什么问题没有被解决 |
| 影响范围 | 这个问题影响哪些功能、角色、数据、流程 |
| 问题等级 | P1（风险接受表仅用于 P1 风险；P0 不能接受风险，必须修正） |
| 接受原因 | 为什么本期不修——资源限制/优先级/影响可控/后续版本已规划 |
| 接受人/责任人 | 谁接受了这个风险 |
| 后续处理 | 什么时候、以什么方式解决 |
| 截止时间 | 解决的 deadline |
| 记录日期 | 风险接受记录日期 |

## 示例

| 风险编号 | 风险类型 | 未解决问题 | 影响范围 | 问题等级 | 接受原因 | 接受人/责任人 | 后续处理 | 截止时间 | 记录日期 |
|---------|---------|-----------|---------|---------|---------|-------------|---------|---------|---------|
| RA001 | 数据口径 | 异常关闭率暂未区分不同异常来源 | 看板统计结果可能与人工统计存在差异 | P1 | 本期先满足单一口径统计，来源拆分进入后续版本 | 产品负责人 | 下版本补充来源维度 | 2026-07-31 | 2026-06-25 |
| RA002 | 历史数据 | 存量模板不做迁移 | 老模板无法展示新增字段 | P1 | 存量模板数量少，可由管理员手动维护 | 业务负责人 | 上线前人工补齐 | 2026-07-05 | 2026-06-25 |
| RA003 | 异常处理 | 接口超时自动重试本期不做 | 网络异常时用户需要手动重试 | P1 | 本期优先保证主流程上线 | PM | 后续补自动重试机制 | 待定 | 2026-06-25 |

## 禁止行为

- 不得将 P0 问题填入风险接受表——P0 必须修正。
- 不得在接受原因中写"先这样"——必须写清楚为什么接受这个风险是合理的。
- 不得将"后续处理"写成"待定"且不留截止时间——待定意味着没有计划。
