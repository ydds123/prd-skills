# 操作流程写作规范

> Belongs to: `05_context/writing-standards/`
> Table contracts: `04_templates/table-templates/table-template-index.json`

所有需要描述用户或系统操作流程的功能统一采用以下结构，不再使用通用五列交互表。

## 1. 主路径（正常路径）

使用从 1 开始的连续编号，按真实发生顺序描述角色动作和系统结果。主路径只写正常完成过程，不把异常塞进步骤正文。

## 2. 分支与异常表

在对应步骤后放置分支与异常表，使用 contract `exception_handling_table`。每行只表达一个可达条件及其处理结果；提示语必须说明原因和下一步行动。

## 3. 关键决策点表

仅当判断会导向不同业务结果时使用 contract `key_decision_table`。普通必填、格式校验、纯技术错误和过细页面校验不属于关键决策点。没有关键决策时不创建空表。

## 4. 多端和多角色

同一操作在不同端的路径一致时合并描述并标明端；路径不同时分别描述。跨角色主线可以先用业务时序图确认，详细操作仍按本规范展开。

## 5. 完成检查

- 主路径是否从入口写到业务结果？
- 分支是否业务可达、符合 MECE，且没有把多个独立条件混在一行？
- 提示语是否写清原因和下一步行动？
- 关键决策是否确实产生不同业务去向？
- 是否残留五列交互表或与主路径重复的描述？
