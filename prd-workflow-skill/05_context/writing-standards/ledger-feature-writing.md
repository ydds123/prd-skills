# 台账型功能写作规范

> Belongs to: `05_context/writing-standards/`
> Load when: 功能以记录维护、查询、配置、导入或导出为主，或台账型能力与业务流程混合出现
> Version: v1.0.0
> Operation flow: `operation-flow-writing.md`

## 核心原则

台账型功能的目标不是把页面控件写全，而是把一个业务记录从“如何找到、如何维护、如何校验、如何批量处理、如何追溯”写成研发和测试可直接执行的契约。

只写有证据且本期支持的能力。原型中的字段、状态、按钮和示例值均是证据候选，不自动等于正式需求；用户最新确认和现有系统真实能力优先于历史材料与示意图。

## 线上与线下边界

先标记每个动作的执行载体：

| 类型 | 写法 |
|---|---|
| 线上动作 | 写清角色、入口、系统处理、数据结果和反馈 |
| 线下动作 | 可以保留业务背景和责任人，但不为纯线下动作虚构页面、实体、状态或异常分支 |
| 线上登记线下结果 | 明确线下事实发生在何处、谁在线上登记、系统保存什么结果 |

## 章节骨架

每个台账型功能按以下顺序组织，只保留本功能实际支持的章节：

1. 页面定义：维护对象、记录颗粒度、角色、数据来源、本期边界。
2. 查询条件：字段、组件、匹配方式、选项或数据来源。
3. 列表字段：字段含义、数据来源、展示和空值规则。
4. 新增/编辑表单：字段、组件、必填、引导文案、字段规则、表单内容规则。
5. 业务规则：唯一性、关联、覆盖、历史影响等产品判断。
6. 操作流程：按“新增/编辑 → 批量导入 → 查询/查看 → 启用/停用 → 导出 → 删除”排序，仅写本功能支持的操作。
7. 权限、全局校验、异常、验收和自测映射。

“字段规则”描述数据来源、联动关系、单位和选项范围；“表单内容规则”描述允许输入或选择什么、什么不合法以及长度/格式约束。两列不得混写。

每个表单字段必须先匹配 `component-specifications.json` 中的一个组件。组件有 `semantic_profiles` 时，再按字段业务含义选择字段语义；组件行为与字段语义共同生成“表单内容规则”。例如“多行文本输入框”只决定允许换行、空格和空行如何处理，200/500/1000 字符由简短备注、处理意见或长正文等字段语义决定。组件列出的 `must_specify` 不得遗漏；有业务证据覆盖默认值时写明最终规则，不在 PRD 中展示组件编号。

## 操作流程结构

每项操作均遵循 `operation-flow-writing.md` 的统一结构。本文件只补充台账特有的操作顺序、导入契约和可达状态规则，不再单独定义另一套流程结构。

## 可达状态原则

异常必须由入口、组件、数据来源或并发时序实际可达：

- 页面下拉只提供满足准入条件的数据时，不重复编写用户正常操作无法选中的状态异常。
- 导入文件、外部同步和开放接口可绕过页面控件，应按其输入边界执行独立校验。
- 不使用“有效、合理、异常”等未定义状态词；状态必须有明确字段、枚举或判断条件。
- 纯网络、接口参数、数据库和过细控件校验默认留到页面/接口设计；只有会影响业务数据是否写入、业务状态或恢复方式时才进入 PRD。

## 分支与异常 MECE

一行原则上只表达一个触发条件。仅当多个条件处于同一校验阶段、处理规则和提示完全一致、且业务上无需区分时才可合并；合并时显式标出差异字段。

分支表至少回答：触发条件、处理规则、提示语/行动建议。不得把多个不同根因压缩成“无法匹配”“状态异常”等不可验收表述。

## 批量导入契约

支持批量导入时，至少说明：

1. 文件名称、格式、数据 Sheet、说明 Sheet、隐藏选项 Sheet（如有）、表头行和数据起始行。
2. 字段矩阵：示例数据、填写方式、必填、格式、值域/主数据、关联、状态和导入处理。
3. Excel 控件：只有适合有限枚举的字段使用下拉；自由文本或需匹配大量主数据的字段按业务证据决定，不照搬页面控件。
4. 校验顺序：必填 → 类型/格式 → 值域/主数据 → 关联 → 状态。同一字段命中前序错误后不重复输出后序错误；同行不同字段错误可一次展示。
5. 写入原子性：明确按文件、按行还是按字段写入；明确部分成功、覆盖和失败数据的处理方式。不得让研发自行决定。
6. 失败信息载体：导入预览、页面结果或失败文件三者选其一或组合，必须有明确依据，不默认创造下载文件。
7. 示例值：来自字典或配置的示例在模板生成时从当前数据动态选取，避免固定示例随字典变化失效。
8. 上传校验时点：动态选项和字典以上传时当前数据为准；旧模板失效时提示下载最新模板。

## 动态选项与提示语

字典、配置或接口返回的选项不得硬编码进错误提示语。提示语优先采用：

`位置 + 填报值（必要时） + 失败原因 + 可直接执行的下一步`

例如：`第{行号}行{字段}“{填报值}”未在当前可选内容中找到，请下载最新模板并重新选择后导入。`

提示语保持短而可行动：

- 用户能自行修正时，只给直接修改动作。
- 主数据维护、字典维护等升级路径统一放在表后作为兜底，不在每条提示中重复。
- 只有同名记录、系统侧冲突等用户无法自行处理的情况，才在单条提示中保留精简联系路径。
- 提示示例不得依赖可能变化的真实字典值；可使用模板生成变量或中性示意值。

## 表单与导入同源校验

新增/编辑和导入对同一业务字段必须共享业务合法性规则，但允许输入控件不同。修改字段长度、数值格式、选项来源或唯一性规则时，至少同步检查：

- 新增/编辑表单；
- 保存时分支与异常；
- 导入模板字段矩阵；
- 导入校验与提示；
- 全局校验规则；
- 验收标准和自测用例。

## 完成检查

- 是否只写本期真实支持的操作？
- 操作顺序是否稳定且没有空章节？
- 表单内容规则是否把合法与非法内容写清？
- 每个表单字段是否匹配组件，并补齐该组件全部 `must_specify`？
- 文本组件是否同时应用组件行为和字段语义，而不是只写最大字符数？
- 分支是否业务可达且符合 MECE？
- 导入原子性、覆盖和失败信息载体是否明确？
- 动态字典是否未被固定提示语或示例写死？
- 提示语是否说明原因和直接行动但不过度展开？
- 页面、导入、校验、验收是否使用同一业务规则？
- 线下动作是否没有被虚构为线上能力？
