# 门禁与复盘

## 可写状态门禁

仅当以下全部满足时放行：

- 背景可被理解
- 真实的用户或角色问题明确
- 本期包含与本期不包含已足够显式
- 关键上下游依赖已识别
- 关键决策已被确认或标记为待确认
- 不存在能推翻 PRD 主方向的问题

失败动作：

- 停止 PRD 写作
- 问最多 3 个优先级最高的问题
- 解释每个答案能解锁哪部分输出

## 决策确认门禁

仅当以下满足时放行：

- 重要的权衡由用户材料确认，或
- AI 的推荐被标记为推荐默认，而非事实

失败动作：

- 产出决策账本
- 标记待确认决策
- 不把待确认决策藏在流畅的文字里

## 第一阶段确认门禁

仅当以下满足时放行：

- 范围稳定
- 主流程稳定
- 涉及角色稳定

失败动作：

- 仅输出第一阶段（v0）
- 请求确认后再进入第二阶段

## 评审阻塞门禁

P0 永远阻塞最终输出。

P1 默认阻塞，除非 PM 明确接受风险，且 PRD 记录在案：

- 已接受的风险
- 原因
- 归属人
- 跟进条件

P2 和 P3 不阻塞最终输出。

## 最终输出门禁

在声称"终版 PRD"之前，检查：

- 无未标记的假设
- 无未经验证的数据被当作事实呈现
- 无虚构的系统能力
- 无混入当前范围的将来规划
- 无技术实现细节（除非被要求）
- 无视觉设计参数（除非被要求）
- 无未解决的 P0
- 无未被接受的未解决 P1

### 可执行内容质量门禁

人工检查必须落为可复验的证据链，不能只在对话中声明“已检查”：

1. `06-内容质量审查.json` 覆盖当前 checklist 全部条目，并记录适用性、结果、证据或原因。
2. `06-审核报告.md` 提供可读的问题说明与最小修改集。
3. `scripts/prd-content-gate.py seal` 重新计算 P0/P1 阻断公式，并对最终 PRD、审查记录、审核报告和 checklist 生成哈希凭证。
4. `scripts/prd-content-gate.py validate` 在最终输出前复验凭证。

以下任一情况阻塞最终输出：

- checklist 条目遗漏、重复或仍为待检查/待补充。
- 通过项没有具体位置和通过依据。
- P0 未修复，或被错误标记为风险接受。
- P1 未修复且没有完整风险接受记录。
- 编写与审查使用同一上下文标识。
- PRD、审查记录、审核报告或 checklist 在封印后发生变化。
- 门禁凭证不存在、结论不是 `pass`，或校验脚本返回非零。

详细契约见 `05_context/prd-standards/content-quality-gate.md`。hook 只能作为第二道保险；工作流内联校验始终是必选步骤。

## Skill 更新门禁

复盘可以提议对可复用资产的改动，但未经人类确认，不得应用。

有效的补丁分类：

| 失败类型 | 补丁目标 |
|---|---|
| 缺业务知识 | Context |
| 缺方法规则 | Skill |
| 缺输出槽位 | Template |
| 缺阻塞条件 | Gate |
| 缺可比案例 | Example |
| 偶发问题 | 仅记录本次运行历史 |

补丁提案必须包含：

- 观察到的失败
- 本次运行的证据
- 为什么现有规则未能拦截
- 提议的目标文件或分类
- 有边界的改动描述
- 回归风险
- 采纳建议

被拒绝的补丁信号：

- 把一次性的偏好变成通用规则
- 扩大 PRD 范围而非修复清晰度
- 复制已有规则
- 让入口 Skill 变重但不提升可靠性
- 没有证据支撑的规则变更

## 内容一致性修复边界

> 适用于 Node 4.5 Content Consistency Sweep。定义 PRD 内部不一致的修复权限。

### 可自动修

满足以下全部条件时可以立即应用修复，不需要 PM 确认：

- 能从 PRD 已有内容中找到明确依据（后期细节 §8-12 > 前期总论 §2-5）。
- 不新增产品事实——只是让已有的判断在多处保持一致。
- 不改变已确认的产品判断。
- 不扩大或缩小需求范围。
- 属于：术语统一、旧说法替换、表格补齐（已有规则同步到验收/自测）、编号修正、引用修正、版本号同步。

**后期细节优先原则**：当后期章节（§8-12）的细则与前期章节（§2-5）的结论冲突时，以后期细节为准——细则是决策的执行结果，总论是执行后的总结。自动更新总论使其匹配细节。

### 需 PM 确认后修

涉及以下内容时必须向 PM 确认，不能自动修：

- 本期做什么 / 不做什么的范围判断
- 角色是否有权限做某操作
- 状态是否可逆
- 默认值如何取
- 异常后是否可恢复
- 数据口径如何计算
- 风险是否接受

### 禁止自动修

以下内容绝对不能自动修：

- 未确认业务规则补成确定事实
- 未确认系统能力补成"本期已支持"
- 未确认外部依赖补成本地实现
- 未确认数据来源补成具体系统 / 表
- P0 问题的风险接受
- Skill 可复用规则修改（此类问题应进入复盘沉淀建议）

## 复盘确认→写入闭环

复盘的最后一步不是输出建议文件，而是逐条确认并写入。未经过这一步的复盘是未完成的复盘。

### 执行规则

1. 复盘 Agent 输出 `08-Skill复盘沉淀建议.md` 后，**必须**逐条询问用户是否采纳。
2. 逐条询问时，每条 patch 给出目标文件路径和改动摘要，用户回答"是/否/改"。
3. 用户确认采纳的 patch，Agent **立即**写入对应目标文件。不应延迟到"下次再做"。
4. 用户拒绝的 patch，在 `08-Skill复盘沉淀建议.md` 中标记"已拒绝"和拒绝原因。
5. 用户要求修改的 patch，Agent 提出修改版本，二次确认后写入。

### 确认格式

```text
Patch 1/7: 端范围一致性检查（缺门禁）
  目标文件: 03_gates/ 或 workflow-protocol.md §4
  改动: 增加规则——若用户确认 Web+App 双端，Review 时分别校验两端可见、详情、处理入口、验收标准是否一致
  采纳？(是/否/改)
```

### 写入后的收尾

所有采纳的 patch 写入完成后，Agent 在 `08-Skill复盘沉淀建议.md` 末尾追加：

```md
## 采纳记录

| 时间 | Patch | 目标文件 | 动作 |
|------|-------|---------|------|
| 2026-06-25 | 端范围一致性检查 | workflow-protocol.md §4 | 已写入 |
| 2026-06-25 | 查询条件固定字段规范 | writing-standards/table-format-conventions.md | 已写入 |
```

### 禁止行为

- 不得在用户确认前写入任何 skill 文件。
- 不得批量询问（"以上 7 条都采纳吗？"）——必须逐条确认。
- 不得因"改动太小"跳过确认步骤。

## 复盘触发器边界

> 复盘触发器（Retrospect Trigger Detector）在用户指正、节点完成、修订完成、内容一致性回扫后自动运行。它可以记录观察、标记复盘候选、建议进入 Skill 复盘，但不得自动修改任何 Skill 可复用文件。

### 触发器可以做的事

- 自动捕获用户指正信号和高风险信号。
- 自动写入 `09-run-log.md` 用户指正记录和复盘触发状态。
- 自动标记 `needs_retrospect_candidate`（T2）或 `needs_retrospect`（T3）。
- T3 时自动生成 `08-Skill复盘沉淀建议.md`。

### 触发器不能做的事

- 不得自动修改 PRD 正文。
- 不得自动修改任何 Skill 可复用文件。
- 不得把用户一次性偏好直接写成全局规则。
- 不得绕过用户确认修改 Skill 文件。
- 不得批量询问"以上全部是否采纳"。
- 不得因为问题严重就自动应用 patch。
- 不得替代 V3.3 gate rules 作为放行标准。
- T3 强制生成的是复盘**建议**，不是复盘**执行**。

### 与复盘确认→写入闭环的关系

触发器只负责"捕捉 → 标记 → 建议"。一旦进入"修改 Skill 文件"阶段，立即受本文件"复盘确认→写入闭环"约束。两条链路是串联的，不是并行的。
- 不得因"改动太小"跳过确认步骤。
